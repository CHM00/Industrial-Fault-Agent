# 工业故障诊断 AI Agent — 期末大作业说明文档

> 基于 LangGraph 构建的具有 RAG 双擎检索、自主审计、人机协同特性的工业故障诊断 Agent

---

## 一、项目概述

### 1.1 项目背景

工业生产中，设备故障诊断高度依赖工程师的个人经验和大量纸质/SOP文档。传统的人工诊断方式存在以下问题：

- **经验依赖强**：资深工程师的经验难以快速传承
- **知识检索慢**：面对新型号设备的故障，查阅手册和资料耗时
- **流程不规范**：不同工程师可能给出不同的诊断步骤，缺乏标准化

本项目利用 LangGraph 框架构建了一个**工业故障诊断 AI Agent**，通过 RAG（检索增强生成）技术融合内部标准 SOP 和外部互联网资讯，自动生成结构化的故障排查流程图，并通过自主审计和人机协同机制确保输出质量。

### 1.2 核心功能

| 功能 | 说明 |
|------|------|
| **RAG 双擎检索** | Milvus 向量库内部 SOP + Tavily 互联网外部搜索，并行并发检索 |
| **知识过滤融合** | LLM 自动清洗外部搜索噪音，以内 SOP 为主、外搜索为辅的分层融合策略 |
| **Mermaid 流程图生成** | 输出工业标准的故障排查流程图，支持可视化渲染 |
| **自主审计机制** | Agent 生成流程图后自动审计，检查逻辑缺口和遗漏步骤 |
| **人机协同** | 审计发现问题时逐个中断，收集专家反馈后修订流程图 |
| **并发与循环** | Fan-out/Fan-in 并发检索 + 逐问循环 + 修订循环 |
| **全链路可观测** | 集成 Langfuse 实现全流程追踪 |

## 三、核心模块详解

### 3.1 知识库构建（Milvus RAG）

**文件**：`sop_documents.py`、`init_knowledge_base.py`

**功能**：将 26 条工业故障诊断 SOP 文档向量化后存入 Milvus 向量数据库。

**流程**：
1. 定义 collection schema：id、vector(2560维)、title、content、category
2. 使用 SiliconFlow 的 `Qwen3-Embedding-4B` 模型对每条 SOP 生成 2560 维向量
3. 批量插入 Milvus，创建向量索引和 category 倒排索引
4. 验证检索效果

**SOP 覆盖领域**（26条）：

### 3.2 LangGraph Agent 核心

**文件**：`fault_agent.py`

#### 3.2.1 State 定义

```python
class AgentState(TypedDict):
    messages: Annotated[List[BaseMessage], add_messages]  # 4种消息类型全覆盖
    fault_input: str                      # 用户输入的故障描述
    internal_knowledge: str               # Milvus 检索到的内部 SOP
    external_knowledge: str               # Tavily 搜索到的外部资讯
    filtered_context: str                 # 分层融合后的上下文
    mermaid_diagram: str                  # 生成的Mermaid流程图
    audit_result: str                     # 审计结果描述
    audit_questions: List[str]            # 审计发现的问题列表
    has_gaps: bool                        # 是否存在缺口（条件路由）
    revision_count: int                   # 修订次数（防死循环，最大3）
    auto_mode: bool                       # 是否自动模式
    current_question_idx: int             # 当前审计问题索引
    expert_feedbacks: List[str]           # 专家反馈列表
    thinking_framework: str               # 审计推理框架（缓存复用）
```

#### 3.2.2 10个功能节点详解

| 序号 | 节点名 | 职责 | 写入State字段 | 消息类型 |
|------|--------|------|-------------|---------|
| 1 | `dispatch` | 入口分发，注入系统角色 | messages | SystemMessage |
| 2 | `search_agent` | 程序化创建tavily_search tool_calls | messages | AIMessage |
| 3 | `retrieve_internal` | Milvus向量检索内部SOP | internal_knowledge, messages | AIMessage |
| 4 | `tools_node` | ToolNode执行tavily_search | messages | ToolMessage |
| 5 | `filter` | 分层融合内外知识 | filtered_context, external_knowledge, messages | AIMessage |
| 6 | `generate` | 生成Mermaid流程图 | mermaid_diagram, messages | AIMessage |
| 7 | `audit` | 自主审计流程图完整性 | has_gaps, audit_questions, audit_result, messages | AIMessage |
| 8 | `evaluate_questions` | 初始化逐问迭代 | current_question_idx, expert_feedbacks, messages | AIMessage |
| 9 | `ask_expert` | 逐问收集专家反馈 | current_question_idx, expert_feedbacks, messages | HumanMessage |
| 10 | `refine` | 结合反馈修订流程图 | mermaid_diagram, revision_count, messages | AIMessage |

#### 3.2.3 条件路由（3条）

| 路由函数 | 位置 | 条件 | 去向 |
|----------|------|------|------|
| `check_gaps` | audit 后 | revision_count≥3 或 has_gaps=False | END |
| `check_gaps` | audit 后 | has_gaps=True | evaluate_questions |
| `more_questions_router` | ask_expert 后 | 还有未回答问题 | ask_expert（Loop） |
| `more_questions_router` | ask_expert 后 | 所有问题已回答 | refine |

### 3.3 并发检索（Concurrency）

**设计**：`search_agent` 节点同时分发到 `retrieve_internal` 和 `tools_node`，两者并发执行，最终在 `filter` 节点汇合（Fan-out / Fan-in）。

```
search_agent ──┬──→ retrieve_internal (Milvus) ──┐
               └──→ tools_node (Tavily) ─────────→ filter (Fan-in)
```

**并发安全**：每个节点只写入自己负责的 State 字段，不互相覆盖。

### 3.4 知识分层融合策略

`filter_knowledge_node` 是并发检索的合并点，采用分层融合策略：

1. **权威性判断**：内部 SOP 权威性 > 外部搜索
2. **冲突处理**：内外矛盾时以内 SOP 为准
3. **补充策略**：内 SOP 不覆盖时用外搜索填补
4. **降噪**：剔除外部信息中的广告、重复、无关内容
5. **标注**：外部补充内容标注「参考」

**四种覆盖场景**：

| 场景 | 内部SOP | 外部搜索 | 处理方式 |
|------|---------|---------|---------|
| 内外都有 | ✅ | ✅ | 以内SOP为主框架，外搜索补充细节 |
| 仅内部有 | ✅ | ❌ | 仅基于内部SOP输出 |
| 仅外部有 | ❌ | ✅ | 基于外部信息，标注「未经现场验证」 |
| 内外都无 | ❌ | ❌ | 依赖LLM常识生成，标注「缺乏参考资料」 |

### 3.5 自主审计机制

`audit_diagram_node` 使用结构化推理框架（MCP Sequential Thinking 等效），从 6 个维度审查流程图：

1. 检查流程图是否有入口节点
2. 检查每个判断节点是否有「是」/「否」两个分支
3. 检查异常路径是否完整
4. 检查流程图是否有终止节点
5. 检查是否有遗漏的关键诊断步骤
6. 检查是否符合工业现场实际操作顺序

### 3.6 人机协同（逐问中断）

当审计发现逻辑缺口时，Agent 进入逐问收集专家反馈模式：

1. **审计发现问题**：列出所有缺口（最多3个）
2. **逐个中断**：每次只展示一个问题 + 当前流程图
3. **收集反馈**：支持人工输入或自动模式（LLM自动生成）
4. **修订流程图**：结合所有反馈，对当前流程图做最小化修改
5. **重新审计**：修订后回到 audit 节点重新审查

### 3.7 云端大模型通信

| 服务 | 提供方 | 用途 |
|------|--------|------|
| LLM | SiliconFlow (DeepSeek-V3) | 所有节点的核心推理能力 |
| Embedding | SiliconFlow (Qwen3-Embedding-4B, 2560维) | 知识库向量化 |
| 外部搜索 | Tavily API | 获取最新故障维修资讯 |

## 八、创新点与亮点

1. **RAG 双引擎并发检索**：内部 Milvus SOP 与外部 Tavily 搜索并行执行，兼顾权威性与时效性
2. **知识分层融合**：提出「内 SOP 为主、外搜索为辅」的分层融合策略，解决内外信息冲突、重复、互补问题
3. **自主审计 + 人机协同**：Agent 生成流程图后自动审计，发现问题逐问中断收集专家反馈，修订后重新审计，形成质量保障闭环
4. **4种 MessageState 全覆盖**：SystemMessage、HumanMessage、AIMessage、ToolMessage 均在图中有明确产生位置
5. **双循环结构**：逐问循环（ask_expert→ask_expert）+ 修订循环（refine→audit→evaluate_questions→ask_expert→refine）
6. **Langfuse 可观测性**：全链路追踪所有 LLM 调用，便于调试和分析

## 十、总结

本项目基于 LangGraph 构建了一个完整的工业故障诊断 AI Agent，实现了从故障输入到流程图输出的完整逻辑闭环。项目通过 RAG 双引擎并发检索、知识分层融合、自主审计和人机协同等设计，在保证输出质量的同时，充分展示了 LangGraph 的并发（Concurrency）和循环（Loop）能力。项目在复杂度、功能完整性和工程实践方面均远超参考案例 Drafter Agent，满足并超过了期末大作业的所有要求。
