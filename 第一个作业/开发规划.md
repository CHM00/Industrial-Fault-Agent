# Airbnb评论主题提炼 Agent 开发规划

## 一、作业需求梳理

### 核心功能
用 LangGraph 设计一个 AI Agent，逐行读取 `reviews.csv` 中的评论文本，通过 LLM 提炼民宿主题标签，支持人机交互修改，最终将评论+标签保存到本地 txt 文档。

### 具体要求（来自"用LangGraph设计AI Agent实现Airbnb评论主题提炼.docx"）

1. **逐行读取** reviews.csv 中的评论文本，发送给云端 AI
2. AI 从评论中**提炼主题标签**（如"干净舒适"、"设施齐全"、"宠物友好"等）
3. 人类可以通过**对话修改**标签（如发出"删除宠物友好标签"指令）
4. 通过 `@tool` 装饰器设计 **"主题标签列表更新器"** —— 支持增/删/改标签
5. 人类确认后（输入"保存"指令），将评论和主题标签列表**保存到本地 txt 文档**
6. 通过 `@tool` 装饰器设计 **"txt文档内容写入工具"**
7. 只需分析 **20 条评论**
8. 可参考 Drafter Agent.ipynb 的设计模式

### 期望输出格式示例

```json
{
  "I stayed with Ed and Ann for two months during a summer internship and their place was a great find - it's very clean, safe, cozy and conveniently located. I could use the kitchen and they added a mini fridge to the room not long after I arrived, which was super helpful. Plus their two dogs are so cute!",
  ["干净舒适", "设施齐全"]
}
```

> 注意：原文中的"宠物友好"被人工删除了，因为宠物友好通常指可以带宠物入住，而评论中说的是房东养了可爱的小狗。这体现了人机交互修改的价值。

## 二、架构设计

### 2.1 状态定义（AgentState）

```python
class AgentState(TypedDict):
    messages: Annotated[Sequence[BaseMessage], add_messages]  # 对话消息历史
    review_index: int          # 当前处理的评论索引（0-19）
    current_review: str        # 当前评论原文
    current_tags: list[str]     # 当前评论的主题标签列表
    saved_results: list[dict]   # 所有已保存的评论+标签结果
```

> 相比 Drafter Agent 的单字段状态 + 全局变量，我们将核心数据纳入 State 管理，避免全局变量问题。

### 2.2 工具定义（@tool）

| 工具 | 功能 | 参数 | 说明 |
|------|------|------|------|
| `update_tags` | 主题标签列表更新器 | `action: str`（"add"/"remove"/"modify"）, `tag: str`, `new_tag: str`（可选） | 增/删/改当前标签 |
| `save_review` | txt文档内容写入工具 | `filename: str` | 将当前评论+标签追加写入txt文件，标记当前评论处理完成 |

### 2.3 节点定义（5个Node）

| 节点 | 功能 | 说明 |
|------|------|------|
| `review_loader` | 评论加载器 | 从CSV读取下一条评论，构建系统提示词，重置标签列表 |
| `topic_analyzer` | 主题分析器（LLM节点） | 接收评论文本，调用LLM提炼或修改主题标签 |
| `tools` | 工具执行器（ToolNode） | 执行 update_tags 或 save_review 工具调用 |
| `review_checker` | 评论完成检查器 | 当前评论保存后，检查是否还有更多评论需要处理 |
| `result_saver` | 最终结果输出 | 所有评论处理完后，打印/输出最终结果 |

### 2.4 图结构与边（含双层Loop）

```
START → review_loader → topic_analyzer → [should_use_tools]
                                                │
                                    有tool_calls ──→ tools ──→ should_continue
                                                              │              │
                                                    已保存(→下条) │       │ 继续修改
                                                                  │              │
                                                                  ↓              ↓
                                                          review_checker    topic_analyzer
                                                           │           │
                                                     还有评论      全部完成
                                                         ↓              ↓
                                                   review_loader    result_saver → END
```

**Loop 结构**：
- **内循环**：`topic_analyzer → tools → topic_analyzer`，人类可反复修改标签直到满意后保存
- **外循环**：`review_loader → ... → review_checker → review_loader`，逐条处理评论

**条件路由**：
- `should_use_tools`：LLM 是否调用了工具？有→tools，无→继续（topic_analyzer）
- `should_continue`：刚执行的是 save_review 还是 update_tags？save→review_checker，update→topic_analyzer

## 三、核心交互流程（单条评论）

```
1. review_loader: 加载第 i 条评论，构建 SystemMessage 注入评论内容
2. topic_analyzer: LLM 分析评论，提炼初始标签 → 调用 update_tags 工具
3. tools: 执行 update_tags，更新 current_tags
4. topic_analyzer: 向人类展示标签，等待反馈
   - 人类说"删除宠物友好标签" → LLM 调用 update_tags(action="remove", tag="宠物友好")
   - 人类说"保存" → LLM 调用 save_review(filename="review_results")
5. tools: 执行对应工具
6. should_continue: 
   - update_tags → 回到 topic_analyzer 继续修改
   - save_review → 进入 review_checker
7. review_checker: 是否还有评论？
   - 是 → 回到 review_loader 加载下一条
   - 否 → 进入 result_saver
8. result_saver: 输出最终结果，结束
```

## 四、关键设计决策

### 4.1 避免 Drafter Agent 的全局变量问题
Drafter Agent 用全局变量 `document_content` 管理文档内容。我们改用 State 中的 `current_tags` 和 `saved_results`，数据流完全在 Graph 内管理。

### 4.2 条件边替代无条件边
Drafter Agent 的 `the_agent → tools` 为无条件边，LLM 不调用工具时也会进入 ToolNode。我们改为条件边：有 `tool_calls` 才进入 tools。

### 4.3 双层循环
- 内循环：单条评论的标签反复修改
- 外循环：多条评论逐条分析
- 使 Agent 具有批处理能力

### 4.4 交互模式
- 优先使用 `input()` 实现真实人机交互
- 也提供"自动模式"选项：LLM 直接保存初始标签，跳过人工修改，方便演示 20 条评论

## 五、开发步骤

### Step 1：环境准备
- 安装依赖：`langgraph`, `langchain`, `langchain-openai`, `python-dotenv`
- 配置 `env.env` 文件（复用现有 API_KEY 和 API_BASE）

### Step 2：CSV数据读取
- 读取 `reviews.csv`，提取前 20 条评论的 `Comments` 列
- 处理编码问题（测试发现 CSV 含非 UTF-8 字符，需用 `errors='replace'` 读取）

### Step 3：状态与工具定义
- 定义 `AgentState`（5个字段）
- 定义 `update_tags` 和 `save_review` 两个 @tool 工具

### Step 4：节点函数实现
- `review_loader`：加载下一条评论，构建 SystemMessage
- `topic_analyzer`：调用 LLM 分析/修改标签（含 system prompt 设计）
- `should_use_tools`：条件路由——是否有工具调用
- `should_continue`：条件路由——是 update_tags 还是 save_review
- `review_checker`：检查是否还有更多评论
- `result_saver`：格式化输出最终结果

### Step 5：图构建与编译
- 添加所有节点到 StateGraph
- 设置边（含条件边）
- 编译图

### Step 6：运行入口与测试
- 实现 `run_review_agent()` 函数
- 用 `app.stream()` 流式执行
- 打印最终 txt 文件内容

### Step 7：Notebook 整理
- 将代码组织到 Jupyter Notebook 的各 Cell 中
- 添加 Markdown 说明
- 展示执行结果，print 最终形成的 txt 文档内容

## 七、与 Drafter Agent 的对比

| 维度 | Drafter Agent | 本项目 Agent |
|------|---------------|-------------|
| 节点数 | 2（the_agent, tools） | 5（review_loader, topic_analyzer, tools, review_checker, result_saver） |
| 工具数 | 2（update, save） | 2（update_tags, save_review） |
| Loop数 | 1（内循环） | 2（内循环修改标签 + 外循环逐条评论） |
| 状态字段 | 1（messages） | 5（messages, review_index, current_review, current_tags, saved_results） |
| 条件路由 | 1（should_continue） | 3（should_use_tools, should_continue, review_checker） |
| 全局变量 | 有（document_content） | 无（全部在State中管理） |
| 终止判断 | 脆弱（字符串匹配） | 鲁棒（基于工具名称和状态字段） |
| MessageState类型 | 3种 | 4种（System/Human/AI/Tool 全覆盖） |
