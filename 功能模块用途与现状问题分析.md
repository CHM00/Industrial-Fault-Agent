# 功能模块用途与现状问题分析

> 修复状态（2026-07-17）：本文第 1～10 节记录的是修复前问题。项目已经完成两轮整改：第一轮解决受控 SOP 与 Milvus 联动，第二轮解决设备上下文、测点治理、SOP 适用范围、资产履历和任务中心闭环。具体状态见文末“修复实施记录”和“其他问题整改矩阵”。

## 1. 分析结论

当前页面并不是单纯的“聊天 + Milvus 知识库”，而是在原有 Milvus SOP 检索之外，又增加了设备资产、结构化诊断、受控 SOP、历史案例和诊断任务生命周期等 P1 功能。用户难以理解的主要原因不是这些功能完全没有实现，而是**页面名称和提示没有说明数据流向，且系统中存在两套彼此独立的 SOP 存储与检索机制**。

最重要的结论如下：

1. 页面“SOP 知识库”上传的文件**不会写入 Milvus**。
2. 上传的 `.md` 原文件也**不会原样保存到本地文件夹**。服务读取文件字节、提取文本后就丢弃原始字节，只把文件名、全文、版本和文本分块写入 SQLite。
3. 页面上传的受控 SOP 实际保存在 `output/runtime/pilot.sqlite3` 的 `knowledge_documents`、`knowledge_versions` 和 `knowledge_chunks` 表中，并用确定性文本相似度检索。
4. 原有 Milvus 知识库是另一套数据源，集合名为 `industrial_fault_knowledge`，主要由 `sop_documents.py` 中的内置 SOP 通过 `init_knowledge_base.py` 初始化。
5. 设备资产和红框中的字段确实会进入诊断上下文，也会参与受控 SOP/历史案例检索；但目前没有明显的页面反馈，因此用户无法判断它们是否生效。
6. 诊断任务中心是异步任务、人工反馈、安全审批、流程审批、重试、归档和导出的管理入口，不是另一个诊断算法。当前英文状态、缺少操作引导和大量遗留测试任务使它很难理解。

## 2. 红框中各字段是做什么的

这些字段用于把一句自然语言故障描述补充成较完整的“设备 + 现场”诊断上下文。

| 字段 | 设计用途 | 当前代码中的实际作用 | 使用示例 |
| --- | --- | --- | --- |
| 设备 | 指定本次故障属于哪台登记设备 | 读取设备编号、名称、类型、厂商、型号、序列号、固件、位置和关键度，加入大模型诊断上下文；检索历史案例时，同一设备的案例额外加权 | 选择 `CHEM-P-2025-003 离心泵` |
| 报警码 | 提供控制器、变频器或设备显示的代码 | 加入受控 SOP 和历史案例的检索查询，并加入大模型上下文 | `F4`、`E101` |
| 运行工况 | 说明故障发生时的负载、环境和复现条件 | 加入检索查询和大模型上下文 | `80% 负载，环境 42℃，启动 20 分钟后出现` |
| 测点 JSON | 提供可计算、可区分名称的现场测量值 | 前端先校验 JSON，再把内容加入检索查询和大模型上下文 | `{"轴承温度":85,"振动":6.2,"出口压力":0.35}` |
| 近期维修 | 告诉系统故障是否可能与换件、检修或参数变更有关 | 进入“结构化设备与现场上下文”，并间接进入受控 SOP/案例检索 | `3 天前更换联轴器并重新对中` |

代码路径为：

`static/app.html` 收集字段 → `web_server.py::_prepare_product_context()` 组装检索查询 → `web_server.py::_structured_context()` 生成结构化文本 → `fault_agent.py::filter_knowledge_node()` 把结构化文本交给大模型融合。

### 2.1 这些字段现在的局限

- 所有字段本质上仍是文本上下文，没有设备测点模型、单位校验、正常范围、时间序列或异常趋势计算。
- “测点 JSON”只保证是合法 JSON，不校验字段名称、单位和取值是否合理。
- 选择设备后，页面不会把厂商、型号等信息展示或回填到输入区，用户看不到“设备选择已产生什么影响”。
- 资产的 `criticality` 当前只进入提示词和指标记录，**不会直接提高确定性安全门等级**。安全门目前只扫描主故障描述中的高压、高温、旋转、易燃等关键词。
- 原始 Milvus 检索仍只使用主故障描述 `fault_input`，不会使用设备型号、报警码和测点；这些结构化字段主要影响后来增加的受控 SOP/历史案例检索及知识融合。

因此，这些字段不是必须全部填写，但信息越具体，受控知识检索和最终流程图越有机会贴近现场。若只是测试通用问题，可以只填主故障描述；若用于真实设备诊断，建议至少选择设备并填写报警码、关键测点和复现工况。

## 3. SOP 知识库到底做什么

SOP 是 Standard Operating Procedure，即标准作业程序。这个模块的目标是让企业自己的操作规程成为诊断时优先级最高、可追溯的依据，避免模型只依靠通用知识或互联网内容回答。

一份受控 SOP 上传后，系统会：

1. 解析 txt、md、csv、tsv、json、pdf、docx、xlsx 或 xlsm 文件中的文本；
2. 计算全文校验和，防止同一版本重复上传；
3. 保存文档版本，更新时将旧版本标记为 `superseded`；
4. 将文本按约 900 字符、120 字符重叠切分；
5. 依据租户、角色权限、文档状态和当前有效版本进行检索；
6. 在诊断时把命中的 SOP 与 Milvus SOP、历史案例共同交给 Agent；
7. 在生成流程后尝试把流程节点映射回 SOP 分块，形成证据引用。

因此它不只是“放文件”，还承担版本、权限、启停、归档和证据追溯功能。

## 4. 为什么在 Milvus 中看不到页面上传的内容

### 4.1 根本原因：项目有两套知识库

| 知识源 | 存储位置 | 写入入口 | 检索方式 | 当前用途 |
| --- | --- | --- | --- | --- |
| 原始 Milvus SOP | 远程 Milvus/Zilliz，或未配置远程时的 `output/milvus_local.db` | 运行 `init_knowledge_base.py`，读取 `sop_documents.py` | Embedding + Milvus IP 向量检索 | 提供内置标准 SOP |
| 页面“受控 SOP 知识库” | `output/runtime/pilot.sqlite3` | 页面“解析并增量索引”按钮 | SQLite 读取有效分块 + Python 文本相似度评分 | 企业文档版本、权限、启停和引用 |

`fault_agent.py::retrieve_internal_node()` 最终会把两类结果拼接起来，所以它们在“诊断效果”上被合并了，但在“存储”上没有合并。

页面按钮写的是“解析并增量索引”，容易让人以为它会生成 Embedding 并写入 Milvus；实际上 `web_server.py::api_import_knowledge_document()` 调用的是 `BusinessStore.import_document()`，该调用链没有任何 PyMilvus 写入操作。

### 4.2 本地 Markdown 文件存放在哪里

当前没有本地上传文件目录。上传链路是：

`浏览器文件字节` → `extract_text()` 提取文本 → `BusinessStore.import_document()` 写 SQLite。

服务端没有执行 `open(..., "wb")` 或其他原文件落盘操作。因此：

- `source_filename` 只保存原文件名；
- `knowledge_versions.content` 保存解析后的全文；
- `knowledge_chunks.content` 保存检索分块；
- 原始 `.md` 文件的字节、原目录结构和附件样式不会被保留。

这意味着当前可以恢复“解析后的文字”，但不能把数据库内容视为原始文件的完整备份。对 PDF、Word 和 Excel 尤其如此，版式、图片、批注、公式等信息可能无法恢复。

### 4.3 当前环境的实际数据

对当前 `output/runtime/pilot.sqlite3` 的只读检查结果为：

- 受控知识文档：1 份；
- 知识版本：1 个；
- 知识分块：1 个；
- 文档标题：`时序大模型`；
- 原文件名：`用于时序的预训练大模型.md`；
- 设备资产：3 台。

这与截图内容一致，说明该文档确实上传成功，只是写入了 SQLite 而不是 Milvus。

### 4.4 另一个可能造成误判的问题

Milvus 连接优先级是：

1. `MILVUS_HOST`；
2. `URL` 指定的 Milvus/Zilliz URI；
3. 本地 `output/milvus_local.db`。

当前 `.env` 同时存在远程连接相关配置，而代码会优先采用 `MILVUS_HOST`。如果运维人员查看的是 `URL` 对应的 Zilliz 实例、另一个数据库或另一个集合，也可能看不到 `industrial_fault_knowledge`。不过这只影响原始 Milvus SOP；无论查看哪个 Milvus 实例，都不会看到页面上传的受控 SOP，因为上传逻辑根本未写 Milvus。

## 5. 设备资产模块用于什么

设备资产是“设备主数据”，作用是避免每次诊断都重复输入设备身份信息，并把多次诊断归属到同一台设备。

当前已实现的作用包括：

- 保存设备编号、名称、类型、厂商、型号、序列号、固件、位置、关键度和状态；
- 在新诊断页面提供设备下拉选择；
- 把所选设备资料加入大模型的结构化上下文；
- 保存诊断案例时记录 `asset_id`；
- 后续检索历史案例时，对同一设备案例增加 `0.2` 相似度权重；
- 在质量成本指标中按设备记录诊断数据；
- 支持设备启用、停用和租户隔离。

### 5.1 当前还没有实现的资产能力

- 没有设备详情页、设备履历时间线和该设备历次故障列表；
- 没有从 CMMS、EAM、ERP、SCADA、PLC 或 IoT 平台同步资产和实时测点；
- 没有“设备型号 → 适用 SOP”的硬过滤关系；
- 没有资产层级，如工厂、车间、产线、机组、子部件；
- 没有基于关键度自动调整审批、安全策略或排障优先级；
- 页面新增设备表单没有暴露后端已支持的序列号、固件和 metadata 字段；
- 停用设备只是不能出现在默认选择列表中，不代表相关任务或案例被禁用。

所以当前设备资产的真实成熟度是“结构化上下文与案例归属”，还不是完整的设备资产管理系统。

## 6. 诊断任务中心有什么能力

任务中心用于管理一次诊断从创建到结束的生命周期。它解决的是以下问题：

- 诊断耗时较长时，记录排队和运行状态；
- Agent 需要专家补充信息时暂停，而不是丢失上下文；
- 高风险场景需要专家明确批准后再继续；
- 生成的流程需要版本化、批准和发布；
- 失败、取消或进程重启后的任务可以定位和重试；
- 任务可以导出 Word、PDF 和现场检查单；
- 保存负责人、租户、时间、错误码和审计事件。

### 6.1 状态说明

| 英文状态 | 中文含义 | 为什么会出现 | 用户应做什么 |
| --- | --- | --- | --- |
| `draft` | 草稿 | 点击“将当前输入保存为草稿” | 打开详情后启动，或归档 |
| `queued` | 排队中 | 已提交，等待并发槽位 | 等待或取消 |
| `running` | 运行中 | Agent 正在检索、生成或审计 | 等待或取消 |
| `waiting_feedback` | 等待专家反馈 | 自动审计发现问题，协同模式暂停 | 回到当前诊断交互回答问题；任务中心详情目前不能直接回答 |
| `waiting_safety` | 等待安全审批 | 主故障描述命中高风险规则 | expert 角色批准或拒绝 |
| `pending_approval` | 等待流程审批 | Agent 已生成结果和初始流程版本 | 详情 → 流程版本 → 批准 → 发布 |
| `completed` | 已完成 | 已批准的流程被发布 | 查看、导出或归档 |
| `failed` | 失败 | 队列超时、模型/检索/流程异常等 | 查看事件和日志后重试 |
| `cancelled` | 已取消 | 用户取消任务 | 可重试或归档 |
| `interrupted` | 进程中断 | 服务重启时任务仍在运行 | 重试 |
| `denied` | 安全审批拒绝 | 专家拒绝高风险诊断 | 修改输入或安全条件后重试 |
| `expired` | 已过期 | 超过会话有效期 | 重试 |
| `archived` | 已归档 | 人工归档或草稿启动后旧草稿归档 | 只作为历史记录 |

### 6.2 为什么当前任务中心让人看不懂

1. 状态直接显示内部英文枚举，没有中文名称、颜色语义和下一步提示。
2. `pending_approval` 的处理入口藏在“详情 → 流程版本”中，页面没有直接显示“请批准并发布”。
3. `waiting_feedback` 的任务详情没有“继续回答”按钮，恢复操作主要依赖原聊天页面中的当前会话。
4. 页面没有状态筛选、搜索、分页、批量归档和清理测试数据功能。
5. 列表只显示故障、负责人和时间，没有设备、报警码、当前步骤、失败原因和等待对象。
6. 多次测试会产生大量同名任务，用户难以区分哪一条是当前任务。
7. 创建草稿时前端只保存故障、研究模式、设备、报警码和工况，**没有保存测点 JSON 与近期维修字段**，导致从草稿启动的信息可能少于直接诊断。
8. 当前数据库中有 27 条任务：`waiting_feedback` 10 条、`pending_approval` 5 条、`cancelled` 6 条、`completed` 6 条。大量等待态和测试故障正是截图显得混乱的直接原因。

## 7. 根因归纳

用户产生疑问的根因可以归为四类：

### 7.1 概念命名问题

- “SOP 知识库”没有说明这是“SQLite 受控知识库”，容易与项目宣传中的 Milvus 知识库混为一谈。
- “解析并增量索引”暗示向量索引，实际只是文本解析、分块和 SQLite 入库。
- “设备资产”听起来像完整 EAM，实际只是设备主数据和诊断上下文。
- “任务中心”没有解释它负责暂停、审批、重试和发布，而不是提供另一套诊断功能。

### 7.2 可见性问题

- 上传后看不到存储后端、分块内容、检索测试结果或是否进入 Milvus。
- 选择设备、填写测点后，没有“将用于本次诊断”的上下文预览。
- 诊断结果页面没有醒目标明本次命中了哪些 Milvus SOP、受控 SOP 和历史案例。

### 7.3 架构并存问题

- 老的 Milvus 向量知识库与新的 SQLite 受控知识库并存，但没有统一管理界面。
- 两套知识库的查询字段也不一致：Milvus 只查询主故障描述，受控库查询更完整的结构化上下文。
- 本地文件、业务数据库和向量数据库分别承担不同职责，但产品界面没有解释这种分层。

### 7.4 产品闭环不完整

- 原文件未保留，缺少下载、预览和重新解析能力。
- 受控 SOP 没有向量化，不适合大规模或语义表达差异较大的知识库。
- 资产关键度与安全门没有形成确定性联动。
- 等待反馈和等待审批状态缺少直接、明确的操作按钮。

## 8. 建议的改进方案

### P0：首先消除误解

1. 将页面名称改为“受控 SOP 文档库”，增加说明：`文件全文和分块保存在业务数据库；当前不会同步到 Milvus`。
2. 将按钮改为“解析并写入受控知识库”，不要使用容易误解的“增量索引”。
3. 上传完成后显示：存储后端、文档 ID、版本、分块数、全文字符数和“测试检索”按钮。
4. 为输入字段增加帮助提示，并增加“本次诊断上下文预览”。
5. 任务状态全部显示中文，并为每种等待状态显示唯一的下一步操作。
6. 为任务中心增加状态筛选、故障/设备搜索和批量归档测试任务。
7. 修复保存草稿时遗漏 `measurements` 和 `maintenance_history` 的问题。

### P1：统一知识库体验

推荐采用“原文件存储 + SQLite 元数据 + Milvus 向量分块”的统一架构：

```text
上传文件
  ├─ 原文件存储：output/knowledge/{tenant}/{document_id}/v{version}/原文件
  ├─ SQLite：文档、版本、权限、状态、校验和、文件路径
  └─ Milvus：每个有效分块的 embedding + document_id/version_id/tenant_id/min_role/status 元数据
```

更新或停用文档时，根据 `document_id + version_id` 删除或屏蔽旧向量。诊断检索只走统一接口，由接口完成租户、权限、状态和版本过滤。原有 `sop_documents.py` 内置知识也应通过同一导入接口迁入，而不是继续维护特殊初始化路径。

如果短期内不迁移，也应在页面明确显示两个区域：

- “内置 Milvus SOP”：展示集合连接、实体数、检索测试和初始化状态；
- “企业受控 SOP”：展示 SQLite 文档、版本、权限、分块和文本检索结果。

### P2：让设备资产真正参与诊断治理

1. 建立 SOP 适用范围字段：厂商、设备类型、型号、固件版本和生效日期。
2. 检索时先做适用范围过滤，再做语义相似度排序。
3. 让资产关键度参与确定性安全规则，例如 `critical` 设备至少进入专家复核。
4. 建立设备详情页，显示历次任务、确认根因、维修结果和常见故障。
5. 定义测点模板与单位，例如温度 `℃`、振动 `mm/s`、压力 `MPa`，并支持正常范围和趋势数据。

## 9. 推荐用户操作方式

在当前代码不修改的前提下，建议按以下方式使用：

1. 把页面 SOP 知识库理解为“受控文本知识库”，不要去 Milvus 中查找页面上传记录。
2. 要查看上传内容，可查询 `output/runtime/pilot.sqlite3`，重点查看 `knowledge_versions.content` 和 `knowledge_chunks.content`。
3. 要初始化或查看原有 Milvus SOP，检查集合 `industrial_fault_knowledge`，并确认实际使用的是 `MILVUS_HOST`、`URL` 还是本地 milvus-lite。
4. 诊断具体设备时选择设备，并填写报警码、带单位的关键测点、故障发生工况和最近变更。
5. 协同模式产生 `waiting_feedback` 后，应在当前聊天诊断界面回答专家问题；`pending_approval` 则进入任务详情的流程版本页面批准并发布。
6. 定期归档或清理测试任务，避免任务中心被重复测试记录占满。

## 11. 修复实施记录（2026-07-17）

本次已完成以下改造：

1. 新增 `managed_knowledge_vector.py`，受控 SOP 当前有效分块同步到 Milvus 集合 `managed_sop_knowledge`。
2. Milvus 分块包含 `tenant_id`、`document_id`、`version_id`、版本号、最低角色、标题、位置和正文，可进行租户与权限过滤。
3. SQLite 继续作为文档、版本、权限和状态的权威数据源，Milvus 定位为可重建的派生语义索引。
4. 新上传原文件保存在 `output/knowledge/{tenant}/{document_id}/v{version}/`，并支持从版本页面下载。修复前上传的历史文件没有原始字节，无法反向恢复，只能保留解析文本。
5. 上传新版本会删除该文档旧向量并写入新版本；停用或归档会移除向量；重新启用或点击“重新同步”会重建。
6. 诊断和知识测试接口优先查询 Milvus；连接、Embedding 或同步异常时自动回退 SQLite，并返回明确告警。
7. 页面改名为“受控 SOP 文档库”，显示 Milvus 集合、同步状态、错误、测试检索和原文件入口。
8. 任务中心状态改为中文，并对等待反馈、等待安全审批和等待流程审批显示下一步说明。
9. 保存诊断草稿时补齐 `measurements` 和 `maintenance_history`。
10. 已把当前数据库中已有的“时序大模型”文档补建到 `managed_sop_knowledge`，实测查询“时序预训练模型”返回 1 条 Milvus 命中，相关度为 `0.7092`。

保留两个 Milvus 集合是有意设计：`industrial_fault_knowledge` 是旧的内置 SOP 集合；`managed_sop_knowledge` 是具备租户、权限和版本元数据的企业受控 SOP 集合。它们在 Agent 内部检索节点统一融合，但不会混用不兼容的集合 schema。
