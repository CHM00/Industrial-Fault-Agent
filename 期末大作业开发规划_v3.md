> **v3 更新说明**：基于 v2 规划与实际代码的差异，重新梳理并发检索与知识平衡策略，修复修订循环闭环、State reducer 类型等问题。

> **v3.1 架构调整**：修订循环恢复到图内，使用 `interrupt()` + `Command(resume=...)` 管理人机交互，避免图重跑导致的 external knowledge 累加问题。图内有两条 Loop，满足作业要求。

## 一、作业要求逐条对照

| 要求 | 本项目如何满足 |
|------|---------------|
| 功能形成逻辑闭环 | 故障输入 → 并发检索 → 知识融合 → 生成流程图 → 审计 → 逐问收集专家反馈 → 修订 → 重新审计 → 输出 |
| 云端大模型通信 | SiliconFlow API（DeepSeek-V3） + SiliconFlow Embedding（Qwen3-Embedding-4B） |
| ≥3种 MessageState | SystemMessage（dispatch）、HumanMessage（用户输入/专家反馈）、AIMessage（各节点LLM输出）、ToolMessage（Tavily @tool 返回）= **4种** |
| ≥4个功能节点 | 10个节点 + 1个ToolNode：dispatch、retrieve_internal、search_agent、tools_node、filter、generate、audit、evaluate_questions、ask_expert、refine |
| ≥1条 Loop 或 Concurrency | **Loop**：逐问循环（ask_expert→ask_expert）+ 修订循环（refine→audit→evaluate_questions→ask_expert→refine）；**Concurrency**：search_agent Fan-out 到 retrieve_internal 和 tools_node |
| 复杂度不低于 Drafter Agent | 10节点+ToolNode + 并发 + 双循环 + 4种Message + RAG双引擎 + 知识分层融合 + 人机协同，远超 Drafter Agent |

**Loops**：
- **逐问循环**：evaluate_questions → ask_expert → more_questions_router → ask_expert（逐个审计问题收集反馈，使用 `interrupt()` 暂停、`Command(resume=feedback)` 恢复）
- **修订循环**：refine → audit → check_gaps → evaluate_questions → ask_expert → refine（修订后重新审计，revision_count 限制3次）

**Concurrency**：search_agent → (retrieve_internal ‖ tools_node) → filter_knowledge（Fan-out/Fan-in）
**ToolMessage**：search_agent 程序化创建 tool_calls → tools_node（ToolNode）执行 tavily_search → 产出 ToolMessage

> **v3.1 架构重构说明**：修订循环从图内移至 `run_fault_agent()` 主要有两个原因：
> 1. **避免图重跑**：此前每次 `app.invoke()` 导致图从 dispatch 重新执行全流程，Tavily 重复搜索使 external knowledge 不断累加
> 2. **稳定流程图**：图重跑会导致流程图完全重新生成而非局部修改，audit 焦点漂移

## 三、并发检索与知识平衡策略（核心设计）

### 3.1 问题背景

并发检索（retrieve_internal 与 search_agent 同时执行）带来了一个关键问题：

> **内部SOP知识与外部搜索结果如何平衡？**
> - 内部SOP：权威、标准化、经过验证，但可能不覆盖所有故障场景
> - 外部搜索：最新、广泛，但可能包含广告/错误/无关信息
> - 两者可能冲突、重复或互补

### 3.2 设计决策：为什么选择并发而非串行

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **串行（当前实现）** | 先内部检索 → LLM判断充分性 → 决定是否外部搜索 | 节省外部搜索API调用 | 额外一次LLM调用判断；无Concurrency边；不符合v2规划 |
| **并发（v3目标）** | 同时检索内部和外部，在filter层做融合 | **有Concurrency边**；节省LLM判断调用；结构更清晰；符合作业要求 | 外部搜索可能浪费（内部已足够时）；filter层逻辑更复杂 |

**结论**：选择并发方案，因为：
1. 作业要求需要 Concurrency 边
2. 省掉 `_judge_internal_sufficiency()` 的LLM调用
3. filter 层做统一融合，逻辑更集中

### 3.3 知识分层融合方案

filter_knowledge_node 是知识平衡的核心，采用 **分层融合**（Layered Fusion）策略：

```
输入：internal_knowledge（Milvus SOP） + external_knowledge（Tavily 搜索）
         │
         ▼
  ┌─────────────────────────────────────────────────┐
  │            filter_knowledge_node                  │
  │                                                  │
  │  1. 权威性判断：内SOP > 外搜索                     │
  │  2. 冲突处理：内外矛盾时以内SOP为准                │
  │  3. 补充策略：内SOP不覆盖时用外搜索填补            │
  │  4. 降噪：剔除外部信息中的广告、重复、无关内容      │
  │  5. 标注：对外搜索来源标注「参考」                  │
  │                                                  │
  │  输出结构：                                       │
  │  ┌─ 故障现象（内SOP 优先，外搜索 补充）           │
  │  ├─ 可能原因（内SOP 为主，外搜索 增补罕见原因）    │
  │  ├─ 诊断步骤（内SOP 标准流程 + 外搜索 最新方法）   │
  │  └─ 处理方案（内SOP 标准方案 + 外搜索 替代方案）   │
  └─────────────────────────────────────────────────┘
         │
         ▼
  filtered_context（结构化诊断参考上下文）
```

### 3.4 四种知识覆盖场景

| 场景 | internal_knowledge | external_knowledge | filter 行为 |
|------|-------------------|-------------------|-------------|
| 内外都有 | ✅ 有相关SOP | ✅ 有搜索结果 | 以内SOP为主框架，外搜索补充细节和最新方案 |
| 仅内部有 | ✅ 有相关SOP | ❌ 无搜索结果 | 仅基于内部SOP输出，标注「无外部补充」 |
| 仅外部有 | ❌ 无相关SOP | ✅ 有搜索结果 | 基于外部信息输出，标注「信息未经现场验证」 |
| 内外都无 | ❌ 无相关SOP | ❌ 无搜索结果 | 依赖LLM常识生成，标注「缺乏专业参考资料」 |

### 3.6 search_agent_node 的并发改造

在串行模式下，`search_agent_node` 需要先等 `retrieve_internal_node` 完成才能判断内部知识是否充分。在并发模式下，`search_agent_node` 无法参考内部检索结果，因此改造为：**基于故障描述本身的特征，由 LLM 决定是否需要搜索外部资料**。

```python
def search_agent_node(state: AgentState) -> dict:
    fault = state["fault_input"]
    core_fault = _extract_core_fault(fault)

    # 并发模式下：不再判断内部充分性，而是基于故障描述决定是否搜索
    prompt = (
        f"当前故障描述：{fault}\n\n"
        f"核心故障关键词：{core_fault}\n\n"
        f"你是一个工业故障诊断助手。请判断是否需要搜索互联网获取该故障的最新诊断信息和解决方案。\n"
        f"以下情况建议搜索：\n"
        f"- 故障涉及特定设备型号的具体报警代码\n"
        f"- 故障现象涉及多个系统的交叉问题\n"
        f"- 用户描述的故障原因不明确，需要更多参考信息\n\n"
        f"如果需要搜索，请使用 tavily_search 工具搜索：'{core_fault} 故障诊断 维修方案'\n"
        f"如果内部知识可能已足够（如常见标准故障），可以不搜索。"
    )
    human_msg = HumanMessage(content=prompt)
    response = llm_with_tools.invoke(state["messages"] + [human_msg])
    return {"messages": [response]}
```

> **设计说明**：并发模式下 `search_agent_node` 不依赖 `internal_knowledge`，仅根据故障描述做决策。这保证了两个分支可以独立执行，不受彼此阻塞。

### 3.7 并发安全：State 字段写入规则

| 节点 | 可写入的 State 字段 | 不可写入 |
|------|---------------------|---------|
| `retrieve_internal` | `internal_knowledge`, `messages` | `external_knowledge` |
| `search_agent` | `messages`（AIMessage） | `internal_knowledge`, `external_knowledge` |
| `tools_node` | `messages`（ToolMessage） | `internal_knowledge`, `external_knowledge` |
| `filter_knowledge` | `filtered_context`, `external_knowledge`, `messages` | — |

> **关键**：`external_knowledge` 的提取不在 `tools_node` 中完成（ToolNode 是 LangGraph 预置节点，不应修改），而是在 `filter_knowledge_node` 中从 `messages` 提取 ToolMessage 内容。

```python
def extract_external_from_tool_messages(state: AgentState) -> str:
    """从 messages 中提取所有 Tavily ToolMessage 的内容"""
    messages = state.get("messages", [])
    tool_contents = []
    for msg in messages:
        if isinstance(msg, ToolMessage) and msg.name == "tavily_search":
            tool_contents.append(msg.content)
    if tool_contents:
        return "\n\n---\n\n".join(tool_contents)
    return ""  # 外部搜索未执行或无结果
```

## 四、State 定义

```python
from typing import Annotated, TypedDict, List, Optional
from langgraph.graph.message import add_messages
from langchain_core.messages import BaseMessage

class AgentState(TypedDict):
    messages: Annotated[List[BaseMessage], add_messages]  # 4种消息类型全覆盖
    fault_input: str                  # 用户输入的故障描述
    internal_knowledge: str           # Milvus 检索到的内部 SOP
    external_knowledge: str           # Tavily 搜索到的外部资讯（由 ToolMessage 提取）
    filtered_context: str             # LLM 分层融合后的上下文
    mermaid_diagram: str              # 生成的 Mermaid 流程图代码
    audit_result: str                 # 审计结果描述
    audit_questions: List[str]        # 审计发现的问题列表
    has_gaps: bool                    # 是否存在缺口（控制 Loop），默认 False
    revision_count: int              # 修订次数（防止无限循环），默认 0
    auto_mode: bool                  # 是否自动模式（跳过人工交互）
    current_question_idx: int        # 当前正在回答的审计问题索引
    expert_feedbacks: List[str]       # 逐问收集的专家反馈列表
    thinking_framework: str           # 审计推理框架（缓存，避免每轮重新生成导致焦点漂移）
```

> **重要**：
> 1. `messages` 使用 `add_messages` reducer（不是自定义 lambda），确保并发分支的消息自动合并
> 2. `revision_count` 和 `has_gaps` 首次 invoke 时必须传入初始值，或在节点函数中用 `state.get("revision_count", 0)` 防御

## 五、10个节点 + 1个ToolNode 详解

### @tool: tavily_search

```python
@tool
def tavily_search(query: str) -> str:
    """搜索互联网获取工业故障维修的最新资讯和解决方案。当内部知识库信息不足时使用此工具补充外部信息。"""
    from tavily import TavilyClient
    client = TavilyClient(api_key=TAVILY_API_KEY)
    results = client.search(query, max_results=3)
    snippets = []
    for r in results.get("results", []):
        title = r.get("title", "")
        content = r.get("content", "")
        url = r.get("url", "")
        snippets.append(f"标题: {title}\n来源: {url}\n内容: {content}")
    return "\n\n---\n\n".join(snippets) if snippets else "未找到相关外部资讯。"
```

> **为什么 @tool 而非普通函数**：只有通过 `@tool` + `ToolNode` 执行，才会产生 `ToolMessage`，满足作业对 ≥3种 MessageState 的要求。Milvus 检索是确定性节点不走 tool_calls，Tavily 作为 LLM 可选调用的工具才走 @tool。

### Node 0: dispatch
- **职责**：入口分发，注入系统角色提示词
- **MessageState**：追加 SystemMessage
- **并发安全**：只写 `messages`，不写 `internal_knowledge` / `external_knowledge`

### Node 1: retrieve_internal
- **职责**：基于 fault_input，从 Milvus 检索内部 SOP 知识
- **输入**：`fault_input`
- **输出**：`internal_knowledge`（str），追加 AIMessage 说明检索结果
- **并发安全**：只写 `internal_knowledge`，不碰 `external_knowledge`

### Node 2: search_agent（程序化工具调用节点）
- **职责**：程序化创建 tavily_search 的 tool_calls，确保每次都执行外部搜索（方案B）
- **输入**：`fault_input`
- **输出**：AIMessage（含 tool_calls 指向 `tavily_search`）
- **设计理由**：并发模式下无法参考内部检索结果判断充分性，且作业要求每次都产生 ToolMessage 保证 Concurrency 行稳定
- **实现**：不调用 LLM，直接从故障描述提取关键词并创建 tool_calls
```python
def search_agent_node(state: AgentState) -> dict:
    fault = state["fault_input"]
    core_fault = _extract_core_fault(fault)
    search_query = f"{core_fault} 故障诊断 维修方案"
    tool_call_id = f"call_tavily_{abs(hash(search_query)) % 1000000:06d}"
    ai_msg = AIMessage(
        content=f"[搜索决策] 需要外部信息补充，搜索关键词：{search_query}",
        tool_calls=[{"name": "tavily_search", "args": {"query": search_query}, "id": tool_call_id, "type": "tool_call"}],
    )
    return {"messages": [ai_msg]}
```
- **输出**：AIMessage（含 tool_calls 指向 `tavily_search`，或纯文本回答）
- **并发安全**：只写 `messages`，不碰 `internal_knowledge` 或 `external_knowledge`
- **并发模式说明**：不再调用 `_judge_internal_sufficiency()`，因为并发模式下内部检索结果尚不可用

### Node 3: tools_node（ToolNode）
- **职责**：执行 `tavily_search` 工具调用，返回 ToolMessage
- **输入**：AIMessage 中的 tool_calls
- **输出**：ToolMessage（搜索结果）
- **MessageState**：ToolMessage（→ **满足作业要求的关键消息类型**）
- **注意**：不直接写 `external_knowledge`，由后续 `filter_knowledge_node` 从 messages 提取

### Node 4: filter_knowledge（知识分层融合节点 — 并发合并点）
- **职责**：LLM 分层融合内外知识，去除广告/无关信息，生成精炼上下文
- **输入**：`internal_knowledge`（来自 retrieve_internal） + 从 messages 提取的 ToolMessage（来自 tools_node）
- **输出**：`filtered_context`、`external_knowledge`
- **MessageState**：AIMessage
- **并发合并**：此节点是 Fan-in 合并点，等待 retrieve_internal 和 search_agent→tools_node 两条分支都完成
- **核心逻辑**：实现 3.5 中定义的分层融合 Prompt

### Node 5: generate_diagram
- **职责**：基于 filtered_context，生成 Mermaid 流程图代码
- **输入**：`filtered_context` + `fault_input`
- **输出**：`mermaid_diagram`
- **MessageState**：AIMessage

### Node 6: audit_diagram（最后一个图内节点）
- **职责**：LLM 自主审计流程图，检查是否有逻辑缺口、遗漏步骤
- **输入**：`mermaid_diagram` + `filtered_context`
- **输出**：`has_gaps`（bool）、`audit_questions`（List[str]）、`audit_result`（str）、`thinking_framework`（str）
- **MessageState**：AIMessage
- **输出格式**：严格 JSON，包含 `has_gaps`、`audit_result`、`questions` 三个字段
- **重要**：审计后图直接走到 END，结果返回给 `run_fault_agent()` 处理

### Node 7: evaluate_questions
- **职责**：初始化逐问迭代，展示所有审计问题的概览
- **输入**：`audit_questions`
- **输出**：`current_question_idx = 0`，`expert_feedbacks = []`
- **MessageState**：AIMessage

### Node 8: ask_expert_per_question
- **职责**：使用 `interrupt()` 暂停等待专家反馈，由 `run_fault_agent` 通过 `Command(resume=feedback)` 恢复
- **输入**：`audit_questions[current_question_idx]`、`mermaid_diagram`
- **输出**：`current_question_idx += 1`，`expert_feedbacks` 追加反馈
- **MessageState**：HumanMessage（专家针对当前问题的反馈）
- **交互机制**：`interrupt()` 暂停 → `run_fault_agent` 收集反馈 → `Command(resume=feedback)` 恢复
- **auto_mode**：`run_fault_agent` 中根据 auto_mode 自动生成反馈

### Node 9: refine_diagram
- **职责**：结合专家反馈，在当前流程图基础上做最小化修改
- **输入**：`mermaid_diagram` + `expert_feedbacks`
- **输出**：`mermaid_diagram`（更新后）、`revision_count += 1`
- **MessageState**：AIMessage
- **重要**：refine 后回到 audit（**修订循环闭环**），而非直接 END

## 六、条件路由

### 路由1: check_gaps（audit 后）
```python
def check_gaps(state: AgentState) -> str:
    revision_count = state.get("revision_count", 0)
    has_gaps = state.get("has_gaps", False)
    if revision_count >= 3:        # 最多修订3次，防止死循环
        return "finished"
    if has_gaps:
        return "needs_feedback"
    return "finished"
```

### 路由2: more_questions_router（ask_expert 后）
```python
def more_questions_router(state: AgentState) -> str:
    idx = state.get("current_question_idx", 0)
    questions = state.get("audit_questions", [])
    if idx < len(questions):
        return "next_question"
    return "all_answered"
```

> **注意**：方案B中不再需要 `should_use_tools` 条件路由，因为 search_agent 始终创建 tool_calls，直接连接到 tools_node。

### 并发合并
并发通过 LangGraph 的 **Fan-out / Fan-in** 机制实现：

```
dispatch → search_agent ──┬──→ retrieve_internal ──→ filter
                          └──→ tools_node ──────────→ filter
```

search_agent 固定连接到 retrieve_internal 和 tools_node 两条并发分支，最终在 filter 节点合并。不再需要 `should_use_tools` 条件路由。

> **并发安全保证**：
> - search_agent 只写 `messages`（AIMessage with tool_calls）
> - retrieve_internal 只写 `internal_knowledge` 和 `messages`（AIMessage）
> - tools_node 由 ToolNode 预置节点处理，只写 `messages`（ToolMessage）
> - 三个字段不互相覆盖，`messages` 通过 `add_messages` reducer 自动合并

## 七、图构建代码（v3.1 最终版）

```python
from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode
from langgraph.checkpoint.memory import MemorySaver

workflow = StateGraph(AgentState)

# 添加节点（10个功能节点 + 1个ToolNode）
workflow.add_node("dispatch", dispatch_node)
workflow.add_node("search_agent", search_agent_node)       # 程序化创建tool_calls
workflow.add_node("retrieve_internal", retrieve_internal_node)
workflow.add_node("tools_node", ToolNode(tools_list))       # 执行 @tool，返回 ToolMessage
workflow.add_node("filter", filter_knowledge_node)
workflow.add_node("generate", generate_diagram_node)
workflow.add_node("audit", audit_diagram_node)
workflow.add_node("evaluate_questions", evaluate_questions_node)
workflow.add_node("ask_expert", ask_expert_per_question_node)  # 使用 interrupt() 暂停
workflow.add_node("refine", refine_diagram_node)

# 入口
workflow.set_entry_point("dispatch")

# 并发边（Fan-out）：search_agent → retrieve_internal 和 search_agent → tools_node
workflow.add_edge("dispatch", "search_agent")
workflow.add_edge("search_agent", "retrieve_internal")
workflow.add_edge("search_agent", "tools_node")

# Fan-in：retrieve_internal → filter 和 tools_node → filter
workflow.add_edge("retrieve_internal", "filter")
workflow.add_edge("tools_node", "filter")

# 串行边
workflow.add_edge("filter", "generate")
workflow.add_edge("generate", "audit")

# 审计后路由：有缺口 → 逐问迭代，无缺口 → 结束
workflow.add_conditional_edges("audit", check_gaps, {
    "needs_feedback": "evaluate_questions",
    "finished": END,
})

# 逐问循环（Loop 1）
workflow.add_edge("evaluate_questions", "ask_expert")
workflow.add_conditional_edges("ask_expert", more_questions_router, {
    "next_question": "ask_expert",       # Loop: 还有更多问题
    "all_answered": "refine",           # 所有问题已回答
})

# 修订循环（Loop 2）：refine → audit
workflow.add_edge("refine", "audit")

# 编译 — MemorySaver 支持断点恢复
checkpointer = MemorySaver()
app = workflow.compile(checkpointer=checkpointer)
```

> **v3.1 关键说明**：
> 1. 修订循环（refine → audit）和逐问循环（ask_expert → ask_expert）都在图内，满足 Loop 要求
> 2. `ask_expert` 使用 `interrupt()` 暂停，`run_fault_agent` 通过 `Command(resume=feedback)` 恢复
> 3. `Command(resume=...)` 从断点恢复执行，而不是从头重跑图，避免了 external knowledge 累加
> 4. 两条 Loop + Concurrency Fan-out/fan-in = 远超 Drafter Agent 的复杂度

## 十一、关键设计决策

### 11.1 为什么不使用 langchain_milvus 的 as_retriever
直接用 `pymilvus` + 自定义 embedding 调用，避免版本兼容问题。复用 `init_knowledge_base.py` 中已封装的 `get_embedding()` 和 `search_milvus()`。

### 11.2 为什么 Tavily 用 @tool 而 Milvus 不用
- **Tavily 是外部搜索**：通过 `search_agent` 程序化创建 tool_calls，走 `@tool` → `ToolNode` → `ToolMessage` 流程（方案B：始终搜索）
- **Milvus 是固定流程**：每次故障诊断都必须检索内部 SOP，是确定性节点
- **满足 ToolMessage 要求**：只有 `@tool` + `ToolNode` 才会产生 `ToolMessage`

### 11.3 方案B：为什么 search_agent 始终搜索（而非LLM决策）

| 考量 | LLM决策方案 | 方案B（始终搜索） |
|------|------------|-----------------|
| Concurrency边 | 需要条件路由（should_use_tools） | 直接边，无条件路由 |
| ToolMessage | 不一定产生（LLM可能不调工具） | 始终产生 ✅ |
| LLM调用 | 多1次（LLM判断是否搜索） | 无额外调用 ✅ |
| 图结构 | 有skip_tools分支 | 更简洁 |
| 并发安全 | 需要考虑skip_tools分支 | 无分支，更安全 ✅ |

### 11.4 为什么修订循环需要在图内（v3.1 架构调整）

v3.1 将修订循环恢复到图内，使用 `interrupt()` + `Command(resume=...)` 管理人机交互：

1. **满足作业 Loop 要求**：图内需要有条 Loop 边，`refine → audit → evaluate_questions → ask_expert → refine` 构成完整闭环
2. **`Command(resume=...)` 从断点恢复**：相比之前的 `app.invoke({"messages": [HumanMessage]})` 方式，`Command(resume=...)` 不会重跑全图，避免了 external knowledge 累加
3. **`interrupt()` 比 `interrupt_before` 更精准**：在节点函数内部调用 `interrupt()`，可以精确控制暂停位置和传入的恢复值

修订循环数据流：
```
ask_expert (interrupt暂停) → run_fault_agent收集反馈 → Command(resume=feedback) → 
继续 ask_expert → more_questions_router → all_answered → refine → audit → check_gaps
```

### 11.5 external_knowledge 为什么不在 tools_node 中写入

`ToolNode` 是 LangGraph 预置节点，我们不应修改其行为。`ToolMessage` 已经写入 `messages`，由 `filter_knowledge_node` 中的 `extract_external_from_tool_messages()` 函数从 messages 中提取并写入 `external_knowledge`。这样设计：
1. 保持 ToolNode 行为不变
2. 知识融合逻辑集中在 filter 节点
3. 并发安全：filter 节点在所有并发分支完成后执行

### 11.6 revision_count 防止死循环
审计-修订闭环最多执行3次，防止 LLM 审计永远发现"缺口"导致死循环。

### 11.7 并发节点的 state 写入安全

| 节点 | 写入字段 | 说明 |
|------|---------|------|
| `dispatch` | `messages` | 只注入 SystemMessage |
| `retrieve_internal` | `internal_knowledge`, `messages` | 并发分支1 |
| `search_agent` | `messages` | 并发分支2（AIMessage，可能含 tool_calls） |
| `tools_node` | `messages` | 并发分支2（ToolMessage，由 ToolNode 自动产生） |
| `filter_knowledge` | `filtered_context`, `external_knowledge`, `messages` | Fan-in 合并点 |
| `generate` | `mermaid_diagram`, `messages` | — |
| `audit` | `has_gaps`, `audit_questions`, `audit_result`, `messages` | — |
| `evaluate_questions` | `current_question_idx`, `expert_feedbacks`, `messages` | — |
| `ask_expert` | `current_question_idx`, `expert_feedbacks`, `messages` | — |
| `refine` | `mermaid_diagram`, `revision_count`, `messages` | — |

> **严格遵守**：每个节点只返回自己负责的字段，绝不返回整个 state。

### 11.8 知识平衡的边界情况处理

| 边界情况 | 处理方式 |
|----------|----------|
| Milvus 连接失败 | `search_milvus()` 返回空字符串，`retrieve_internal_node` 输出 "内部知识库未检索到相关SOP信息" |
| Tavily API 不可用 | `tools_node`（ToolNode）会返回错误 ToolMessage，`filter` 节点仅基于内部 SOP |
| Milvus 和 Tavily 都无结果 | `filter` 节点依赖 LLM 常识生成，标注 "缺乏专业参考资料" |
| 内外信息冲突 | 以内部 SOP 为准，在 `filtered_context` 末尾注明存在冲突 |
| 内外信息重复 | 去重后优先保留内部 SOP 版本 |

## 十五、MCP 等效功能集成

> 由于 Windows 环境 npx 安装 MCP server 存在权限问题（EPERM），采用 **Python 原生实现** 三个 MCP 等效功能，功能等价且更可靠。

### 15.1 Sequential Thinking（结构化推理）

**MCP 等效**：`mcp_tools.sequential_think()`

**集成点**：`audit_diagram_node`

**原理**：在审计 Prompt 中注入结构化推理框架，迫使 LLM 按 6 个维度逐步审查流程图：

```
1. 检查流程图是否有入口节点（故障报告作为起点）
2. 检查每个判断节点是否有'是'和'否'两个分支
3. 检查异常路径是否完整
4. 检查流程图是否有终止节点
5. 检查是否有遗漏的关键诊断步骤
6. 检查流程图是否符合工业现场实际操作顺序
```

**代码位置**：`fault_agent.py:291-306`（AUDIT_PROMPT）和 `fault_agent.py:319-325`（audit_diagram_node）
