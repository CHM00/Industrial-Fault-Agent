# 前端界面优化分析

> 分析对象：`static/app.html`（现行主界面，约 94 KB，登录/RBAC、诊断会话、任务中心、设备资产、受控 SOP、流程审批、质量看板）与 `static/index.html`（旧版界面，约 73 KB，仅基础诊断）。
> 结论先行：前端功能覆盖面已经完整，但存在**双前端并存、单文件巨石、若干安全隐患、SSE 无断线恢复、移动端功能缺失**五类系统性问题。建议按文末 P0→P2 路线收敛，不需要一次性重写。

## 1. 现状概述

| 维度 | 现状 |
| --- | --- |
| 形态 | 两个静态 HTML，由 FastAPI 直接挂载，无构建、无框架、零依赖 |
| 样式 | 全部内联在 `<style>`，CSS 变量统一配色，原生 `<dialog>` 做弹层 |
| 逻辑 | 全部内联在 `<script>`，全局 `state` 对象 + 全局函数，字符串模板拼 HTML |
| 通信 | 手写 SSE 解析（`fetch` + `ReadableStream`），REST 用 `apiJson` 封装 |
| 渲染 | Mermaid 本地 vendor（app）/ CDN mermaid@10（index），每次 `mermaid.run` 全量渲染 |

值得肯定的设计：

- `esc()` 转义函数在绝大多数插值点都有使用；`app.html` 的 Mermaid 配置为 `securityLevel:"strict"`。
- 角色权限在 UI 层有降级处理（`can()` 控制按钮显隐/禁用），而非只依赖后端。
- 手写 SSE 解析、原生 `<dialog>`、无框架方案让部署链路极短，与单实例试点定位匹配。
- 诊断会话的状态机（`running` / `pendingKind`）管理清晰，安全审批与专家提问复用同一反馈区但做了文案/权限区分。

## 2. 问题清单

### 2.1 架构与工程化

**A1. 双前端并存且能力不同步（高优先级）**

- `index.html` 是旧版：无登录、无 RBAC、无 P1 全部功能（任务中心/资产/SOP/审批/看板），仍调用 `/api/start` 与 `/api/resume`。
- 它通过 CDN 加载 `mermaid@10`（第 10 行），而 `app.html` 用本地 `/vendor/mermaid`。前后端校验链路（`validate_mermaid.mjs`）固定了 Mermaid 版本，旧版前端渲染版本与后端校验版本不一致，可能出现"后端校验通过但前端渲染失败/样式不同"。
- 第 12–14 行有一个路径判断后 `location.replace("/")` 的自毁逻辑，说明作者本意是下线它，但文件仍可被直接访问。
- 建议：直接删除 `index.html`，或在服务端把 `/static/index.html` 301 到 `/`，避免旧入口被收藏夹/搜索引擎命中。

**A2. 单文件巨石**

`app.html` 一个文件承载 7 个"页面"（会话 + 5 个管理中心弹层 + 登录），其中管理中心全部是字符串模板函数：`openAssets`、`openKnowledge`、`openTaskCenter`、`openProcedureEditor`、`openDashboard` 各自 50–100+ 行的 HTML 拼接。问题：

- HTML 写在 JS 字符串里，编辑器无法高亮/格式化/校验，改一处结构要在字符串里找；
- 每次打开弹层都 `manageBody.innerHTML = 全量模板` 再 `querySelectorAll` 重挂事件，事件绑定与模板强耦合，漏挂即静默失效；
- 无模块边界，无法按需加载（打开"质量看板"的代码在首屏就被解析）。

建议：不引入框架的前提下，先拆成 ES Modules（`static/js/api.js`、`sse.js`、`views/*.js`），`<script type="module">` 原生可跑；长期再评估 Vite + Vue/React。

**A3. 无静态检查与测试**

- 无 ESLint/Prettier/类型检查；`tests/` 目录全部是后端测试，前端零覆盖。
- SSE 事件处理逻辑在 `start()`、`submitFeedback()`、`resumeTaskFromCenter()`、任务中心的"重试"和"启动草稿"五处几乎复制粘贴，事件名拼写错误不会被任何工具发现。
- 建议：至少加一个共享的 `runDiagnosisStream(url, body, progressEl)` 收口五处重复；引入 Playwright 做一条"发起诊断→回答问题→导出 Word"的端到端用例。

### 2.2 安全

**B1. 外部来源 URL 未做协议白名单（真实可利用点）**

`renderResult` 中外部研究来源直接进 href：

```js
`<a href="${esc(s.url||"#")}" target="_blank" ...>`
```

`esc()` 只转义 HTML 特殊字符，**不拦截 `javascript:` 协议**。研究子系统抓回的外部 URL 属于不可信数据（`FILTER_PROMPT` 里都把外部内容标记为 untrusted），若某个来源 URL 为 `javascript:...`，专家点击即执行。建议渲染前校验 `new URL(url).protocol` 仅允许 `http:`/`https:`，否则按纯文本展示。

**B2. `index.html` 的 Mermaid 为 `securityLevel:"loose"` + CDN 无 SRI**

`loose` 会允许图中的 `click` 回调与外部链接执行，配合 CDN 无完整性校验，供应链与注入风险叠加。删除 `index.html`（见 A1）即可一并消除。

**B3. innerHTML 拼接依赖人工转义纪律**

整个文件的安全边界是"每个插值点都记得调 `esc()`"。目前覆盖较好，但这种约定在多人协作下必然破防。建议：

- 新增渲染代码优先用 `textContent` + DOM API，或引入 60 行的 tagged-template 转义助手（`html`...`` 自动转义插值）；
- 服务端为 `/` 与 `/static/*` 增加 CSP 响应头（`default-src 'self'`），把 XSS 影响面再压一层。

**B4. 历史遗留敏感存储痕迹**

第 466 行 `localStorage.removeItem("pilotApiKey")` 是清理旧版把 API Key 存 localStorage 的代码。当前版本已改 HttpOnly Cookie（正确方向），该清理行可保留一个版本周期后删除，同时确认无其他 `localStorage` 敏感写入。

### 2.3 性能

**C1. 流程编辑器每次按键全量渲染 Mermaid（体感最差的一处）**

`openProcedureEditor` 中 `$("procedureSource").oninput = render`，每个按键都触发一次完整 `mermaid.run`。工业流程图节点多，输入到几十个字符时明显卡顿。建议 300–500ms 防抖，并加"手动刷新预览"按钮兜底。

**C2. 每个待审版本渲染一份完整 SVG**

协同模式下每轮修订都会 `addReviewDiagram` 追加一张 review 卡片并立即 `mermaid.run`，3 次修订就是 3 份大 SVG 常驻 DOM，加上最终版图共 4 份。建议：卡片内默认只渲染缩略占位（节点数/字符数摘要），`<details>` 展开或点"全屏查看"时才真正渲染。

**C3. 日志与历史列表无界增长**

- 运行日志每条 SSE `progress` 事件 `appendChild` + 强制 `scrollTop`（同步布局），长任务上千条日志节点全留在 DOM；
- 历史列表 `loadHistory()` 全量拉取全量重排，诊断完成后整个列表推倒重建。

建议：日志保留最近 N 条（如 300 条）滚动裁剪；历史列表加分页或"加载更多"，完成时只做单项插入而非全量重拉。

**C4. Mermaid 渲染结果无缓存**

同一张图在对话卡片、全屏 `diagramDialog`、流程编辑器预览中会反复渲染。可以 `Map<diagramHash, svg>` 做内存缓存，全屏查看时直接复用。

### 2.4 可用性与交互

**D1. SSE 断流即"视图死亡"，无恢复路径（最重要的体验问题）**

`streamSSE` 一旦网络抖动/服务重启/笔记本合盖断流，只走 `error` 回调显示失败。实际上后端任务可能仍在 `running` 或已停在 `waiting_feedback`，但前端没有任何引导用户去任务中心恢复的提示。同样，刷新页面后正在运行的任务无法回到流式视图。建议：

- 断流时捕获任务状态：若 `sessionId` 已知，自动调 `/api/jobs/{id}` 查询，处于 `waiting_feedback/waiting_safety` 则直接重建反馈区；
- 错误提示中给出"前往任务中心恢复"的按钮；
- 长期可改 `EventSource` + `Last-Event-ID`，或服务端在任务记录里持久化已发事件，恢复后补发。

**D2. `alert()` / `prompt()` 阻断式反馈**

错误提示、审批意见（`prompt("审批意见")`）全用原生弹窗，阻断 SSE 期间的交互且样式割裂。建议统一的 toast/内联错误条 + 审批意见内嵌输入框。

**D3. 诊断设置仅存内存，刷新即丢**

`state.settingsSnapshot` 与会话内设置都不持久化。现场工程师常用的设备、研究参数每次重开页面都要重填。建议非敏感设置（设备选择、研究模式、深度、时限、搜索服务）存 `localStorage`。

**D4. 管理中心"整页重载"式更新**

停用设备、重新同步、归档任务等操作成功后都是 `await openXxx()` 整体重取重绘，弹层内容闪动且丢失滚动位置。建议操作后局部更新行内状态。

**D5. 细节问题**

- "＋ 新诊断"是 `location.reload()`（第 451 行），粗暴且丢失未保存输入；
- 时间戳原样输出 ISO 字符串（历史列表、任务表格），无本地化格式化；
- 点击侧边栏按钮到弹层打开之间无任何 loading 反馈（数据 `await` 期间界面静止，弱网下像没点上）；
- 测点表格与"高级 JSON"两个输入源并存，冲突时静默以 JSON 为准，无提示。

### 2.5 可访问性与移动端

**E1. 移动端是"功能缺失"而非"响应式"**

`@media(max-width:820px)` 直接 `display:none` 侧边栏与日志面板，且没有提供任何替代入口——手机上无法访问历史、任务中心、资产、SOP、看板，也无法看运行日志。现场场景恰恰是移动设备高发场景。建议：侧边栏改抽屉（汉堡按钮触发），日志改底部可折叠 sheet。

**E2. 对比度贴线**

`--muted:#746f68` on `--bg:#f7f5f1` 对比度约 4.6:1，12px 小字（`.item-time`、`.user-meta`、`.form-hint`）在强光/投影/工业平板上可读性差。建议正文辅助色加深到 `#5f5a54` 一档。

**E3. 其他**

- 无深色模式（夜班中控室场景有真实需求，CSS 变量已就位，加一套 `prefers-color-scheme` 覆盖成本低）；
- 无键盘导航/快捷键（除 Enter 提交）；`status-pill` 只用 `title` 传达状态，屏幕阅读器不友好；
- 图标按钮部分有 `aria-label`（好），但管理表格内按钮全靠文字大小写区分。

## 3. 优化路线图

### P0 —— 安全与正确性（1 天内可完成）

1. 删除或重定向 `index.html`，统一 Mermaid 版本与 `strict` 模式（解决 A1/B2）。
2. 外部来源 URL 加 `http/https` 协议白名单（B1）。
3. 流程编辑器预览加防抖（C1）。
4. 五处 SSE 处理收敛为一个共享函数，事件名用常量表（A3 前置）。

### P1 —— 核心体验（2–4 天）

5. SSE 断流恢复：已知 `sessionId` 时自动查任务状态并重建反馈区；错误提示带"去任务中心恢复"入口（D1）。
6. toast/内联错误替代 `alert/prompt`；非敏感诊断设置存 `localStorage`（D2/D3）。
7. 日志滚动裁剪 + 历史列表分页；review 卡片懒渲染（C2/C3）。
8. 管理中心操作后局部刷新；加载态骨架；时间本地化（D4/D5）。

### P2 —— 工程化与长期（按迭代排期）

9. 拆 ES Modules，管理中心五个视图分文件；逐步引入模板转义助手与 CSP（A2/B3）。
10. 移动端抽屉导航 + 日志底部 sheet；深色模式（E1/E3）。
11. Playwright 端到端用例接入 CI；视团队情况评估 Vite + 框架迁移的收益。

## 4. 不建议现在做的事

- **不建议立刻全面重写为 React/Vue**：当前无框架方案与"单实例试点 + FastAPI 静态挂载"的部署模型匹配良好，重写收益主要在 P2 阶段才显现，且会引入构建链路与产物缓存问题。
- **不建议上重型组件库**：当前自定义 CSS 变量体系已覆盖全部 UI，引入组件库反而破坏一致的视觉密度。
- **不建议为了"现代"换 WebSocket**：SSE 单向推送与诊断场景完全匹配，问题只在前端断流处理，不在协议。
