# LangGraph 工业故障诊断 Agent 项目功能实现报告

> 报告日期：2026-07-16  
> 分析范围：`LangGraph_Agent_Work` 目录下的 Python 源码、前端页面、部署文件、测试用例及现有说明文档  
> 报告重点：项目整体功能实现、核心流程与各模块职责

## 1. 项目概述

本项目是一个面向工业设备故障排查场景的智能诊断系统。系统以 LangGraph 作为流程编排框架，将工业安全预检、内部 SOP 检索、外部资料研究、大模型知识融合、Mermaid 排障流程生成、结果审计、专家反馈与流程修订组织为可追踪的状态图，并通过 FastAPI、SSE 和 Web 页面向用户提供完整的诊断服务。

项目当前的主要交付物包括：

- 可审计、可修订的 Mermaid 故障排查流程图；
- 包含安全评估、知识来源、审计过程和专家反馈的 Markdown 诊断报告；
- 用于后续查询和案例复用的结构化诊断记忆；
- Word、PDF 诊断文档和 CSV 现场检查单；
- 设备资产、SOP 文档、历史案例、流程版本、任务状态和质量指标等业务数据。

从实现形态看，项目已经由单次模型调用原型扩展为一个具备安全门、知识治理、人机协同、任务治理和业务闭环的单实例试点系统。

## 2. 系统总体架构

系统可以划分为六个层次：

```text
交互层
  Web 页面 / FastAPI / SSE / CLI
       ↓
业务服务层
  任务中心 / 资产管理 / 知识管理 / 案例管理 / 流程审批 / 导出与看板
       ↓
智能编排层
  LangGraph 主诊断图 / 基础研究图 / 主管研究图 / 人工中断与恢复
       ↓
知识与推理层
  Milvus SOP / 受控知识库 / 历史案例 / 外部搜索 / LLM 融合与审计
       ↓
安全与治理层
  工业安全规则 / API Key / RBAC / 租户隔离 / 限流 / 并发 / 取消与重试
       ↓
持久化与交付层
  SQLite Checkpoint / 业务数据库 / Mermaid / Markdown / JSON / DOCX / PDF / CSV
```

推荐的实际运行入口是：

- `fault_agent.py`：核心诊断图、命令行诊断和主要智能能力；
- `web_server.py`：Web 页面、SSE 流式诊断和业务 API；
- `init_knowledge_base.py`：内置 SOP 向量知识库初始化。

`main.py` 包含更广泛的多模态、技能编排、知识图谱、意图识别和离线缓存设想，但它依赖当前目录中未提供的多个模块，因此属于实验性或扩展性入口，不应视为当前可独立部署的主链路。

## 3. 核心诊断流程实现

### 3.1 LangGraph 主流程

`fault_agent.py` 使用共享的 `AgentState` 保存故障输入、研究配置、内外部知识、流程图、审计结果、安全状态、专家反馈、证据映射和 Token 使用事件。主图包含以下节点：

1. `dispatch`：生成请求标识，整理故障描述、资产、报警码、工况、测点和维修历史等结构化上下文。
2. `assess_safety`：调用确定性工业安全规则，对任务进行风险识别。
3. `safety_gate`：高风险任务在任何检索和模型调用之前暂停，等待专家批准或拒绝。
4. `start_research`：初始化内外部研究任务及相关状态。
5. `retrieve_internal`：检索 Milvus SOP、受控知识文档和历史诊断案例。
6. `external_research`：根据配置关闭外部研究，或执行基础/主管研究工作流。
7. `filter`：融合并筛选内部知识、外部证据和安全约束，控制上下文长度。
8. `generate`：由大模型生成 Mermaid 故障排查流程图。
9. `map_evidence`：为流程图节点匹配 SOP、案例或外部来源证据。
10. `audit`：检查流程完整性、逻辑合理性、安全性和信息缺口。
11. `evaluate_questions`：整理需要专家确认的问题。
12. `ask_expert`：自动模式生成反馈，人工模式通过 LangGraph `interrupt()` 暂停并逐题等待专家意见。
13. `refine`：依据专家反馈修订流程图，再次执行证据映射和审计。

内部检索与外部研究采用并行分支，二者完成后汇合到知识过滤节点。审计通过后结束；存在缺口时进入反馈与修订分支，修订最多执行三轮，避免无限循环。

### 3.2 人机协同机制

系统将专家参与实现为真正的可恢复工作流，而不是普通聊天提示：

- 高风险诊断可在安全门节点暂停；
- 审计发现缺口后可逐题暂停；
- Web API 使用 `Command(resume=...)` 将审批或反馈送回原 LangGraph 线程；
- `thread_id` 用于区分不同诊断会话；
- SQLite checkpoint 可在服务或图对象重建后恢复状态；
- 每次专家提问会携带当前 Mermaid 图和修订版本，便于针对具体版本审查。

### 3.3 Mermaid 质量控制

生成和修订后的流程图不会直接保存。`mermaid_pipeline.py` 依次完成：

- 从模型回复中提取 Mermaid 代码；
- 规范换行和部分语法形式；
- 限制最大字符数；
- 拒绝 `click`、JavaScript URL、脚本标签和初始化指令等不安全内容；
- 通过常驻 Node 子进程调用官方 `mermaid.parse()` 校验语法；
- 对可修复的语法错误仅允许一次大模型最小化修复；
- 校验器异常或二次校验失败时阻止流程图进入交付链路。

这一机制保证最终流程图不仅能显示，而且经过了独立于生成模型的确定性语法质量门。

## 4. 各功能模块说明

### 4.1 核心 Agent 模块

| 文件 | 功能实现 |
|---|---|
| `fault_agent.py` | 定义诊断状态、LangGraph 节点和路由；实现安全门、内外检索、知识融合、流程生成、审计、专家反馈、修订、命令行运行及最终保存。 |
| `safety.py` | 使用正则规则识别高压/带电、易燃易爆、高温高压、旋转机械、保护旁路、受限空间和有毒环境；输出风险等级、控制措施、禁止项及是否需要专家审批。 |
| `evidence_mapping.py` | 解析 Mermaid 节点，按关键词重叠度将节点映射到知识文档、历史案例和外部资料，并标记可能存在的证据冲突。 |
| `mermaid_pipeline.py` | 完成 Mermaid 提取、规范化、安全过滤、官方解析器校验、子进程复用和校验器健康检查。 |
| `validate_mermaid.mjs` | Node.js 侧的 Mermaid 官方语法解析入口。 |

### 4.2 外部研究模块

研究子系统位于 `research/`，通过统一数据契约和预算控制屏蔽不同搜索服务的差异。

| 文件/目录 | 功能实现 |
|---|---|
| `research/contracts.py` | 定义研究选项、研究任务、外部来源、任务结果、告警和总体结果；完成 URL 规范化、来源去重和稳定来源 ID 分配。 |
| `research/configuration.py` | 从环境变量和 Agent 状态构造研究配置，统一搜索服务、超时、上下文和密钥配置。 |
| `research/budget.py` | 在研究任务间分配搜索额度，限制总搜索次数，并保证重试同样消耗预算。 |
| `research/gateway.py` | 对外提供统一研究入口，支持 `off`、`basic`、`supervisory` 三种模式，汇总来源、状态、告警、任务结果和实际搜索次数。 |
| `research/basic/graph.py` | 实现“生成查询—网络搜索—来源总结—反思补查”的基础研究图，按研究深度决定循环次数。 |
| `research/supervisory/graph.py` | 将复杂故障拆分为多个研究任务，复用基础研究图并行执行，分析结果后生成综合研究结论。 |
| `research/providers/` | 提供统一搜索供应商抽象及 Tavily、DuckDuckGo、SearXNG、Perplexity 四种实现。 |
| `research/llm.py` | 创建研究专用 LLM，提供异步文本调用和 JSON 结果解析。 |

研究模块还实现了以下控制：

- 单次搜索超时、总研究时限和最大重试次数；
- 最大任务数、总搜索次数、单来源长度和外部上下文总长度；
- 搜索失败、认证缺失、限流和超时等结构化告警；
- 外部网页内容按不可信输入处理，避免其覆盖系统要求；
- 主管研究综合时清除模型生成但未被真实检索结果验证的 URL。

### 4.3 内部知识库模块

| 文件 | 功能实现 |
|---|---|
| `sop_documents.py` | 保存用于知识库初始化的内置工业故障 SOP 文档。 |
| `init_knowledge_base.py` | 创建 `industrial_fault_knowledge` 集合，调用嵌入接口生成 2560 维向量，写入 SOP 并执行检索验证。 |
| `Milvus.py` | 封装 Milvus/Zilliz 连接、集合访问和部分历史兼容逻辑。 |
| `document_ingestion.py` | 安全提取 TXT、Markdown、JSON、CSV、PDF、DOCX、XLSX 文本，限制上传大小和最大提取字符数，并按重叠窗口切分知识块。 |
| `business_store.py` | 实现受控知识文档、版本、角色可见性、生效状态、内容分块及增量索引。 |

运行时内部知识由三类数据共同构成：

1. Milvus/Zilliz 或 milvus-lite 中的向量化标准 SOP；
2. 业务数据库中按租户、角色和生效状态过滤的受控文档；
3. 已回填最终根因和处理结果的历史诊断案例。

Milvus 连接支持远程主机、Zilliz URI 或本地 `milvus-lite` 三种形式；未配置远程服务时使用 `output/milvus_local.db`。

### 4.4 结果保存与文档导出模块

| 文件 | 功能实现 |
|---|---|
| `mcp_tools.py` | 保存 Mermaid、Markdown 报告和 JSON 诊断记忆；报告中记录安全评估、内外部知识、来源链接、研究状态、审计、专家反馈和证据映射；提供本地诊断记忆搜索。 |
| `artifact_export.py` | 将诊断状态导出为 DOCX、PDF 和带 UTF-8 BOM 的 CSV 现场检查单；PDF 支持中文字体选择。 |
| `md_to_word.py` | 提供 Markdown 转 Word 的辅助能力。 |
| `output/diagrams/` | 保存 `*_diagram.mmd` 和 `*_report.md`。 |
| `output/memory/` | 保存结构化诊断历史 JSON。 |

保存前会对外部研究结果做压缩，在保留来源、告警和追溯字段的同时控制历史文件体积。

### 4.5 Web 服务与前端模块

`web_server.py` 是 FastAPI 服务入口，复用一个已编译的 LangGraph 应用，并以 SSE 持续发送节点进度、研究结果、专家问题、流程图版本、错误和最终完成事件。

`static/app.html` 提供单页 Web 工作台，主要实现：

- 故障描述和结构化诊断信息录入；
- 自动/人工反馈模式选择；
- 外部研究模式、深度、供应商、预算和总时限配置；
- SSE 节点状态及研究告警展示；
- 安全审批和专家问题反馈；
- Mermaid 实时预览、源码编辑、全屏查看和复制；
- 诊断历史、任务列表、资产、知识库、流程版本和看板操作；
- 报告、流程图、Word、PDF 和检查单获取。

Web 层主要 API 分组如下：

| API 分组 | 主要能力 |
|---|---|
| `/api/start`、`/api/resume` | 启动诊断、提交安全审批或专家反馈，并通过 SSE 返回执行过程。 |
| `/api/jobs` | 创建草稿、启动、查询、归档、取消和重试诊断任务。 |
| `/api/assets` | 创建设备资产、查询和更新资产信息。 |
| `/api/knowledge` | 上传知识文档、查询版本、变更状态和检索知识。 |
| `/api/cases/{session_id}/confirm` | 回填确认根因、处理方案、可信度和专家评分，使案例可以被后续诊断召回。 |
| `/api/procedures` | 创建流程版本、比较 Diff、逐块采纳、批准、拒绝和发布。 |
| `/api/jobs/{session_id}/export` | 导出 DOCX、PDF 或 CSV 检查单。 |
| `/api/dashboard/quality` | 汇总诊断数量、耗时、Token、估算成本和专家修订等指标。 |
| `/api/history`、`/api/diagram`、`/api/report`、`/api/memory` | 查询历史交付物。 |
| `/health/live`、`/health/ready`、`/metrics` | 提供存活检查、关键依赖就绪检查和运行指标。 |

### 4.6 设备资产与结构化诊断模块

资产模型支持资产编码、名称、类型、厂商、型号、序列号、固件、关键度、位置、自定义元数据和启用状态，并按租户隔离。

一次诊断除自由文本外，还可以接收：

- 设备资产 ID；
- 报警码；
- 运行工况；
- 电压、电流、温度等测量数据；
- 维修历史；
- 附件引用；
- 受控知识和历史案例的最大召回数。

这些字段会被整理为结构化上下文，参与安全评估、知识检索和流程生成，从而减少仅依赖一段自然语言描述导致的信息损失。

### 4.7 业务闭环与流程治理模块

`business_store.py` 基于 SQLite 实现了以下业务实体和生命周期：

- 设备资产：创建、更新、启用/停用及租户隔离；
- 知识文档：上传、版本递增、角色权限、生效/失效和检索；
- 诊断案例：保存诊断状态，回填根因和处理结果后进入可召回案例库；
- 流程版本：保存 Mermaid 草稿，生成版本 Diff，逐块采纳差异；
- 审批状态：支持草稿、批准、拒绝和发布流程；
- 质量指标：记录耗时、Token、估算费用、修订次数和诊断结果。

由此形成如下业务闭环：

```text
创建诊断任务 → 生成排障流程 → 专家修订与审批 → 现场文档导出
      ↑                                              ↓
      └──── SOP/历史案例召回 ← 根因和处理结果回填 ←──┘
```

### 4.8 持久化任务与运行时模块

| 文件 | 功能实现 |
|---|---|
| `checkpointing.py` | 在开发环境使用内存 checkpoint，在生产环境默认使用官方 SQLite Saver；启用 WAL、同步和忙等待配置，生产初始化失败时不会静默降级。 |
| `runtime_store.py` | 保存任务基础信息、状态、所有者、租户、当前节点、错误、过期时间和事件日志；支持任务查询、超时清理和服务重启后的中断状态协调。 |
| `governance.py` | 实现滑动窗口限流和全局诊断并发上限。 |
| `observability.py` | 记录 HTTP、鉴权、限流、诊断和耗时等计数指标，并输出 Prometheus 风格文本。 |

任务状态覆盖草稿、运行、等待安全审批、等待专家反馈、完成、失败、取消和归档等业务阶段。用户可以在任务中心查看历史、取消运行任务或对失败任务发起重试。

### 4.9 身份认证、权限和租户隔离模块

`security.py` 提供 API Key 认证和四级 RBAC：

| 角色 | 典型权限 |
|---|---|
| `viewer` | 查看获授权的任务和交付物。 |
| `operator` | 启动、恢复、取消、重试诊断，维护资产和流程草稿。 |
| `expert` | 管理知识、执行高风险审批、确认案例、审批/发布流程、查看质量看板。 |
| `admin` | 具备最高权限，并可访问系统指标。 |

中间件在业务请求进入路由前完成身份识别、角色校验、租户与主体维度限流，并统一添加内容类型、防嵌套、来源策略和 CSP 等安全响应头。生产环境要求配置有效的试点 API Key；开发环境可使用本地管理员主体便于调试。

### 4.10 可观测性与健康检查

系统提供三类运行状态接口：

- `/health/live`：确认 Web 进程存活；
- `/health/ready`：检查 checkpoint、任务数据库、业务数据库和 Mermaid 校验器等关键依赖；
- `/metrics`：输出请求量、状态码、认证失败、限流和耗时等指标。

同时，Agent 会记录每个主要 LLM 节点的输入、输出和总 Token，并将其汇总到质量成本看板。Langfuse 回调为可选能力，未安装或未配置时不影响主流程运行。

## 5. 数据存储与输出结构

项目采用“数据库状态 + 文件交付物 + 向量知识库”的混合存储方式：

| 存储位置 | 内容 |
|---|---|
| `output/runtime/checkpoints.sqlite3` | LangGraph 线程 checkpoint，用于中断和恢复。 |
| `output/runtime/pilot.sqlite3` | 任务、事件、资产、知识版本、案例、流程审批和质量指标。 |
| `output/milvus_local.db` | 无远程 Milvus 时的本地向量知识库。 |
| `output/diagrams/` | Mermaid 流程图和 Markdown 报告。 |
| `output/memory/` | JSON 诊断记忆。 |

生产容器通过卷挂载 `output/`，保证容器重建后业务数据仍可保留。SQLite 适用于单实例试点；多副本或高并发部署需要迁移到集中式 checkpoint、业务数据库和任务队列。

## 6. 部署与配置实现

项目提供：

- 固定版本的 Python 依赖；
- Mermaid 固定版本的 `package-lock.json`；
- 基于 Python 3.10 的 Dockerfile；
- Node.js、npm 和中文字体安装；
- Docker Compose 端口、环境变量和持久化卷配置；
- 容器级 `/health/ready` 健康检查；
- 生产环境认证、SQLite checkpoint、并发、限流和会话 TTL 配置。

关键外部依赖包括兼容 OpenAI 协议的 LLM/Embedding 服务、可选的 Milvus/Zilliz、可选的外部搜索供应商以及 Node Mermaid 解析器。

## 8. 当前实现边界

虽然项目已经具备较完整的试点能力，但仍存在以下明确边界：

1. SQLite 和进程内会话协调主要面向单实例部署，不适合直接进行多副本水平扩展。
2. 受控知识检索当前以 SQLite 文本相似度和本地分块为主，大规模知识库仍应使用集中式向量索引。
3. 流程编辑以 Mermaid 源码、实时预览和版本 Diff 为主，尚未实现拖拽式图形编辑器。
4. 历史案例相似度实现适合试点数据量，后续应结合向量检索、设备层级和案例可信度排序。
5. 附件字段已进入诊断模型，但图片、音频、振动频谱和实时测点等多模态解析尚未接入主流程。
6. 项目尚未直接连接 OPC UA、MQTT、SCADA、CMMS/EAM 等工业系统。
7. 确定性安全门降低了风险，但系统输出仍属于辅助建议，高风险现场操作必须由具备资质的人员审批和执行。
8. `main.py` 所描述的知识图谱、技能注册、BERT 意图路由、本地模型和离线缓存并非当前主项目的可运行交付能力。

## 9. 项目功能实现总结

项目已经完成以下核心闭环：

```text
结构化故障输入
  → 确定性工业安全预检与专家门禁
  → SOP、受控知识、历史案例和外部资料检索
  → 大模型知识融合与 Mermaid 排障流程生成
  → 官方语法校验、步骤证据映射和独立审计
  → 专家逐题反馈、流程修订、版本 Diff 与审批发布
  → Markdown/JSON/DOCX/PDF/CSV 多格式交付
  → 根因与处理结果回填、案例复用和质量成本统计
```

从功能实现角度看，项目的突出特点不是单纯“调用大模型生成答案”，而是将诊断过程构造成可暂停、可恢复、可审查、可追溯和可治理的业务工作流。当前版本适合在单企业、单工厂或小规模团队中开展受控试点；若要进入多租户、大规模生产阶段，下一步应重点建设集中式数据与任务基础设施、工业系统集成、多模态证据接入以及基于黄金案例的诊断质量评测体系。
