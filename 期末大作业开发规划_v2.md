# 工业故障诊断 AI Agent — 开发规划 v2

## 一、作业要求逐条对照

| 要求 | 本项目如何满足 |
|------|---------------|
| 功能形成逻辑闭环 | 故障输入 → 检索 → 过滤 → 生成流程图 → 审计 → 人工确认/修改 → 输出最终流程图 |
| 云端大模型通信 | SiliconFlow API（DeepSeek-V3），已在 `.env` 中配置 |
| ≥3种 MessageState | SystemMessage（dispatch注入）、HumanMessage（用户输入/专家反馈）、AIMessage（LLM生成）、ToolMessage（Tavily工具返回） |
| ≥4个功能节点 | 8个节点 + 1个ToolNode：dispatch、retrieve_internal、search_agent、tools(ToolNode)、filter_knowledge、generate_diagram、audit_diagram、human_expert、refine_diagram |
| ≥1条 Loop 或 Concurrency 边 | Concurrency：dispatch 同时分发到内/外检索；Loop：audit → human → refine → audit 闭环 |
| 复杂度不低于 Drafter Agent | 8节点+ToolNode + 并发 + 循环 + 4种Message(含ToolMessage) + RAG双引擎 + 人机协同，远超 Drafter Agent |

**Loops**：
- **逐问循环**：evaluate_questions → ask_expert → more_questions → ask_expert（逐个审计问题收集反馈）
- **修订循环**：refine → audit → check_gaps → evaluate_questions → ask_expert → refine（修订后重新审计）

**Concurrency**：dispatch → (retrieve_internal ‖ search_agent→tools_node) → filter_knowledge
**ToolMessage**：search_agent 调用 @tool(tavily_search) → tools_node 执行 → 产出 ToolMessage

## 三、State 定义

```python
from typing import Annotated, TypedDict, List, Optional
from langgraph.graph.message import add_messages
from langchain_core.messages import BaseMessage

class AgentState(TypedDict):
    messages: Annotated[List[BaseMessage], add_messages]  # 4种消息类型全覆盖
    fault_input: str                  # 用户输入的故障描述
    internal_knowledge: str           # Milvus 检索到的内部 SOP
    external_knowledge: str           # Tavily 搜索到的外部资讯（由 ToolMessage 提取）
    filtered_context: str             # LLM 过滤融合后的上下文
    mermaid_diagram: str              # 生成的 Mermaid 流程图代码
    audit_result: str                 # 审计结果描述
    audit_questions: List[str]        # 审计发现的问题列表
    has_gaps: bool                    # 是否存在缺口（控制 Loop），默认 False
    revision_count: int              # 修订次数（防止无限循环），默认 0
    auto_mode: bool                  # 是否自动模式（跳过人工交互）
    current_question_idx: int        # 当前正在回答的审计问题索引
    expert_feedbacks: List[str]       # 逐问收集的专家反馈列表
    thinking_framework: str           # 审计推理框架（缓存，避免每轮重新生成导致焦点漂移）
```

> **重要**：`revision_count` 和 `has_gaps` 首次 invoke 时必须传入初始值，或在节点函数中用 `state.get("revision_count", 0)` 防御。TypedDict 无默认值机制，缺失会 KeyError。

## 四、8个节点 + 1个ToolNode 详解

### @tool: tavily_search
```python
from langchain_core.tools import tool

@tool
def tavily_search(query: str) -> str:
    """搜索互联网获取工业故障维修的最新资讯。"""
    from tavily import TavilyClient
    client = TavilyClient(os.environ["TAVILY_API_KEY"])
    results = client.search(query, max_results=3)
    return "\n".join([r["content"] for r in results.get("results", [])])
```
> **为什么 @tool 而非普通函数**：只有通过 `@tool` + `ToolNode` 执行，才会产生 `ToolMessage`，满足作业对 ≥3种 MessageState 的要求。Milvus 检索是内部固定流程不走 tool_calls，Tavily 作为 LLM 可选调用的工具才走 @tool。

### Node 0: dispatch
- **职责**：入口分发，注入系统角色提示词
- **MessageState**：追加 SystemMessage（"你是工业故障诊断AI助手..."）
- **并发安全**：dispatch 只写 `messages`，不写 `internal_knowledge` / `external_knowledge`，避免与下游并发节点冲突

### Node 1: retrieve_internal
- **职责**：基于 fault_input，从 Milvus 检索内部 SOP 知识
- **输入**：`fault_input`
- **输出**：`internal_knowledge`（str），追加 AIMessage 说明检索结果
- **实现方式**：直接调用 pymilvus 搜索（复用现有 MilvusClass 的 embedding + search）
- **并发安全**：只写 `internal_knowledge`，不碰 `external_knowledge`

### Node 2: search_agent（LLM 决策节点）
- **职责**：LLM 判断是否需要调用 Tavily 搜索外部资讯，如果需要则生成 tool_calls
- **输入**：`fault_input` + 历史 messages
- **输出**：AIMessage（含 tool_calls 指向 `tavily_search`，或纯文本回答）
- **MessageState**：AIMessage（含 tool_calls）

### Node 3: tools_node（ToolNode）
- **职责**：执行 `tavily_search` 工具调用，返回 ToolMessage
- **输入**：AIMessage 中的 tool_calls
- **输出**：ToolMessage（搜索结果），同时将结果写入 `external_knowledge` 字段
- **MessageState**：ToolMessage（→ **这是满足作业要求的关键消息类型**）

> **并发合并问题**：retrieve_internal 和 search_agent→tools_node 并发执行时，两者都对 `messages` 做追加。LangGraph 的 `add_messages` reducer 会自动合并两条分支的消息（按时间戳或内部排序），不会丢失。但 `external_knowledge` 的写入需要在 tools_node 中完成（从 ToolMessage 内容提取），不能依赖 search_agent 节点。

### Node 4: filter_knowledge（原 Node 3）
- **职责**：LLM 过滤融合内外知识，去除广告/无关信息，生成精炼上下文
- **输入**：`internal_knowledge` + `external_knowledge`
- **输出**：`filtered_context`
- **MessageState**：AIMessage
- **关键 Prompt**：
  > "你是一位资深工业故障诊断工程师。请从以下内部SOP和外部资讯中，提取与故障直接相关的诊断步骤和维修方案，去除广告、重复和无关信息，输出精炼的故障诊断参考上下文。"

### Node 5: generate_diagram（原 Node 4）
- **职责**：基于 filtered_context，生成 Mermaid 流程图代码
- **输入**：`filtered_context` + `fault_input`
- **输出**：`mermaid_diagram`
- **MessageState**：AIMessage
- **关键 Prompt**：
  > "请根据以下故障诊断上下文，生成一份工业故障排查 Mermaid 流程图。要求：使用 flowchart TD 格式，包含判断节点、操作步骤、异常路径，节点文本使用中文。"

### Node 6: audit_diagram（原 Node 5）
- **职责**：LLM 自主审计流程图，检查是否有逻辑缺口、遗漏步骤
- **输入**：`mermaid_diagram` + `filtered_context`
- **输出**：`has_gaps`（bool）、`audit_questions`（List[str]）、`audit_result`（str）
- **MessageState**：AIMessage
- **关键 Prompt**：
  > "请审计以下故障排查流程图，检查：1) 是否有逻辑断裂或遗漏步骤；2) 异常路径是否完整；3) 是否需要补充特定设备的操作细节。如果存在问题，请列出具体问题；如果流程图完整无误，请回复'PASS'。"

### Node 7: evaluate_questions
- **职责**：初始化逐问迭代，展示所有审计问题的概览
- **输入**：`audit_questions`
- **输出**：`current_question_idx = 0`，`expert_feedbacks = []`
- **MessageState**：AIMessage（"开始逐个收集专家反馈，共N个问题"）

### Node 8: ask_expert_per_question
- **职责**：逐个展示审计问题并收集专家反馈
- **输入**：`audit_questions[current_question_idx]`、`mermaid_diagram`
- **输出**：`current_question_idx += 1`，`expert_feedbacks` 追加反馈
- **MessageState**：HumanMessage（专家针对当前问题的反馈）
- **interrupt**：`interrupt_before=["ask_expert"]`，图在展示问题前暂停，等待人工输入
- **auto_mode**：自动模式下生成默认反馈，skip `input()`

### Node 9: refine_diagram
- **职责**：结合人类专家反馈，重新生成流程图
- **输入**：`mermaid_diagram` + 专家反馈
- **输出**：`mermaid_diagram`（更新后）
- **MessageState**：AIMessage

## 五、条件路由

### 路由1: check_gaps（audit 后）
```python
def check_gaps(state: AgentState) -> str:
    revision_count = state.get("revision_count", 0)  # 防御 KeyError
    has_gaps = state.get("has_gaps", False)
    if revision_count >= 3:  # 最多修订3次，防止死循环
        return "finished"
    if has_gaps:
        return "needs_feedback"
    return "finished"
```

### 路由2: should_use_tools（search_agent 后）
```python
def should_use_tools(state: AgentState) -> str:
    last_message = state["messages"][-1]
    if hasattr(last_message, "tool_calls") and last_message.tool_calls:
        return "tools"
    return "skip_tools"  # LLM 未调用工具，直接跳到 filter
```

### 路由3: 并发合并
并发通过 `add_edge(["retrieve_internal", "tools_node"], "filter_knowledge")` 实现 fan-in。
> **并发安全**：两个并发节点各自只写自己负责的 state 字段（internal 写 `internal_knowledge`，tools_node 写 `external_knowledge`），不互相覆盖。两者都会追加 `messages`，`add_messages` reducer 会自动合并。

## 六、图构建代码骨架

```python
from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode, tools_condition
from langgraph.checkpoint.memory import MemorySaver

workflow = StateGraph(AgentState)

# 定义 @tool
tavily_search_tool = tavily_search  # 见第四节定义
tools_list = [tavily_search_tool]

# 添加节点
workflow.add_node("dispatch", dispatch_node)
workflow.add_node("retrieve_internal", retrieve_internal_node)
workflow.add_node("search_agent", search_agent_node)       # LLM 决策是否调 Tavily
workflow.add_node("tools_node", ToolNode(tools_list))       # 执行 @tool，返回 ToolMessage
workflow.add_node("filter", filter_knowledge_node)
workflow.add_node("generate", generate_diagram_node)
workflow.add_node("audit", audit_diagram_node)
workflow.add_node("evaluate_questions", evaluate_questions_node)  # 初始化逐问迭代
workflow.add_node("ask_expert", ask_expert_per_question_node)    # 逐个问题收集反馈
workflow.add_node("refine", refine_diagram_node)

# 入口
workflow.set_entry_point("dispatch")

# 并发边
workflow.add_edge("dispatch", "retrieve_internal")
workflow.add_edge("dispatch", "search_agent")

# search_agent → tools_node（有 tool_calls）或直接到 filter（无 tool_calls）
workflow.add_conditional_edges("search_agent", should_use_tools, {
    "tools": "tools_node",
    "skip_tools": "filter",
})
workflow.add_edge("tools_node", "filter")
workflow.add_edge("retrieve_internal", "filter")

# 串行边
workflow.add_edge("filter", "generate")
workflow.add_edge("generate", "audit")

# 审计后路由：有缺口 → 逐问迭代，无缺口 → 结束
workflow.add_conditional_edges("audit", check_gaps, {
    "needs_feedback": "evaluate_questions",
    "finished": END,
})

# 逐问循环
workflow.add_edge("evaluate_questions", "ask_expert")
workflow.add_conditional_edges("ask_expert", more_questions_router, {
    "next_question": "ask_expert",      # Loop: 还有更多问题
    "all_answered": "refine",           # 所有问题已回答，修订流程图
})

# 修订后回到审计（Loop）
workflow.add_edge("refine", "audit")

# 编译 — 必须使用 checkpointer 才能支持 interrupt
checkpointer = MemorySaver()
app = workflow.compile(checkpointer=checkpointer, interrupt_before=["ask_expert"])
```

> **关键说明**：
> 1. `ToolNode` 是 LangGraph 预置节点，自动执行 `@tool` 函数并返回 `ToolMessage`
> 2. `interrupt_before=["ask_expert"]` 在每个审计问题前暂停，实现逐问中断
> 3. `more_questions_router` 检查 `current_question_idx < len(audit_questions)`，形成逐问循环
> 4. 修订后重新审计形成第二条循环：refine → audit → evaluate_questions → ask_expert → refine
> 5. **两条循环**（逐问循环 + 修订循环）+ 并发检索 = 远超 Drafter Agent 的复杂度

### 7.3 Milvus 知识库准备

**统一方案**：使用 `pymilvus` 直接操作（复用已有 `MilvusClass` 的连接和 embedding），**不使用** `langchain_milvus`（避免版本兼容问题）。

需要新建独立的初始化脚本 `init_knowledge_base.py`，职责包括：
1. 连接 Milvus（复用 `MilvusClass.connect_milvus()`）
2. 创建 `industrial_fault_knowledge` collection（专用 schema）
3. 将预置 SOP 文档做 embedding 后批量插入
4. 验证检索效果

collection schema：
```python
fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
    FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=2560),  # Qwen3-Embedding-4B
    FieldSchema(name="title", dtype=DataType.VARCHAR, max_length=256),
    FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=2000),
    FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=64),
]
```

> **为什么不用 langchain_milvus**：已有的 `MilvusClass` 已经封装了 embedding 调用和连接逻辑，且 embedding dim=2560 与 Zilliz Cloud / 自建 Milvus 已对齐。引入 langchain_milvus 可能造成 dim 不匹配或版本冲突。

## 十一、关键设计决策

### 11.1 为什么不使用 langchain_milvus 的 as_retriever
直接用 `pymilvus` + 自定义 embedding 调用，避免 `langchain_milvus` 版本兼容问题。复用已有 `MilvusClass` 的 embedding 方法和连接逻辑。

### 11.2 为什么 Tavily 用 @tool 而 Milvus 不用
- **Tavily 是外部搜索**：LLM 根据故障类型决定是否需要搜索（本地 SOP 足够就不搜），所以它是 LLM 自主决策的工具调用，走 `@tool` → `ToolNode` → `ToolMessage` 流程
- **Milvus 是固定流程**：每次故障诊断都必须检索内部 SOP，不需要 LLM 决定"要不要查内部知识库"，所以 Milvus 检索是确定性节点
- **满足 ToolMessage 要求**：只有 `@tool` + `ToolNode` 才会产生 `ToolMessage`，普通节点函数写 state 字段不会产生任何 Message 类型

### 11.3 human_expert_node 的实现方式
采用 **方案A：LangGraph 的 `interrupt` + `Command(resume=...)`** 机制。

**前置条件（缺一不可）**：
1. 编译时必须传入 `checkpointer=MemorySaver()`，否则 `interrupt_before` 会报错
2. `invoke` 时必须传 `thread_id`：`config={"configurable": {"thread_id": "1"}}`
3. 恢复执行使用 `Command(resume=feedback_text)`

```python
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import Command

# 编译
app = workflow.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["human"]  # 在 human 节点前中断
)

# 第一次执行：运行到 human 节点前自动暂停
result = app.invoke(
    {"fault_input": "PLC通讯故障", "messages": [], "revision_count": 0, "has_gaps": False},
    config={"configurable": {"thread_id": "1"}, "callbacks": [langfuse_handler]}
)

# 人类专家查看审计问题后，输入反馈并恢复执行
result = app.invoke(
    Command(resume="请补充变频器相关判断步骤"),
    config={"configurable": {"thread_id": "1"}, "callbacks": [langfuse_handler]}
)
```

> **Fallback 方案**：如果 `interrupt` 在某些环境不稳定，可改用 `input()` 阻塞：
> ```python
> def human_expert_node(state):
>     feedback = input(f"审计发现以下问题：{state['audit_questions']}\n请输入专家反馈：")
>     return {"messages": [HumanMessage(content=feedback)]}
> ```

### 11.4 revision_count 防止死循环
审计-修订闭环最多执行3次，防止 LLM 审计永远发现"缺口"导致死循环。

### 11.5 内部检索使用已有 Milvus 还是新建 collection
新建 `industrial_fault_knowledge` collection，与已有的 `Food_List` / `User_Memory` 分离，避免数据混淆。

具体做法：在 `init_knowledge_base.py` 中复用 `MilvusClass` 的 `connect_milvus()` 方法和 `embedding()` 方法，但定义独立的 collection schema（见 7.3），写入后通过 `collection.search()` 做向量检索。

`fault_agent.py` 中的 `retrieve_internal_node` 同样复用 `MilvusClass` 的连接和 embedding，查询 `industrial_fault_knowledge` collection。

### 11.6 并发节点的 state 写入安全
当 `retrieve_internal` 和 `search_agent→tools_node` 并发执行时：
- `retrieve_internal` 只写 `internal_knowledge` 和 `messages`（AIMessage）
- `tools_node` 只写 `external_knowledge` 和 `messages`（ToolMessage）
- 两个字段不重叠，不会互相覆盖
- `messages` 通过 `add_messages` reducer 自动合并两条分支的追加，不会丢失

节点函数中**严禁**返回整个 state，只返回自己负责的字段：
```python
# ✓ 正确
def retrieve_internal_node(state: AgentState) -> dict:
    results = search_milvus(state["fault_input"])
    return {"internal_knowledge": results, "messages": [AIMessage(content=f"内部检索到{len(results)}条SOP")]}

# ✗ 错误 — 会覆盖并发分支的 state
def retrieve_internal_node(state: AgentState) -> dict:
    state["internal_knowledge"] = results  # 不要修改整个 state
    return state  # 不要返回整个 state
```

## 十三、生产级优化记录

### 13.1 LLM 调用健壮性
- **问题**：SiliconFlow API 偶发 500/503 错误导致流程中断
- **方案**：新增 `invoke_llm_with_retry()` 函数，指数退避重试（最多5次，间隔 3s→6s→12s→24s→48s）
- **所有 `llm.invoke()` 调用已替换为 `invoke_llm_with_retry()`**

### 13.2 Prompt 长度截断
- **问题**：内外知识拼接后可能超出模型上下文窗口，导致 500 错误
- **方案**：对所有 LLM 调用的输入做截断保护
  - `filter_knowledge_node`: internal 限制 6000 字符，external 限制 6000 字符
  - `generate_diagram_node` / `audit_diagram_node` / `refine_diagram_node`: context 限制 8000 字符
- **调试日志**：`filter_knowledge_node` 中打印内外部知识字符数，便于排查

### 13.3 审计问题数量限制
- **问题**：模型可能输出过多审计问题，导致人机交互轮次过长
- **方案**：
  - Prompt 侧：`AUDIT_PROMPT` 中明确约束 `questions列表最多只能包含3个问题`
  - 代码侧：解析时 `questions[:3]` 截断保底

### 13.4 修订流程图稳定性
- **问题**：修订时 LLM 每次重新创造一个全新流程图，而非在原图基础上做局部修改
- **方案**：强化 `refine_diagram_node` 的 Prompt，明确要求：
  1. 只修改与专家反馈直接相关的节点和连线
  2. 未涉及的节点和连线必须完全保持原样
  3. 不得重新组织、重命名、增删无关部分
  4. 不要改变整体结构和风格，仅做局部修补

### 13.5 审计推理框架缓存
- **问题**：`audit_diagram_node` 每轮都重新调用 `sequential_think()` 生成推理框架，导致审计焦点漂移
- **方案**：在 `AgentState` 中新增 `thinking_framework: str` 字段
  - 首次审计时生成并缓存到 state
  - 后续审计轮次复用缓存的推理框架，确保审计焦点稳定

### 13.6 AUDIT_PROMPT 格式化修复
- **问题**：`AUDIT_PROMPT` 中的 JSON 示例 `{...}` 被 Python `.format()` 误解析为占位符
- **方案**：将 JSON 示例中的 `{` → `{{`，`}` → `}}`，确保 `.format()` 正确解析

### 13.7 PyMilvus 弃用警告抑制
- **方案**：`warnings.filterwarnings("ignore", ...)` 额外抑制 `langchain_openai` 的弃用警告
